The problem no one talks about

Every team I've worked with has the same silent problem. An AI assistant that's brilliant in a vacuum but inconsistent under pressure. One developer uses it to scaffold clean, layered code. Another uses it to dump raw implementation directly into a controller. A third asks it the same architectural question it answered three sessions ago — and gets a different answer.

We adopted Claude Code enthusiastically. Within two months, we had velocity improvements that impressed leadership. We also had PRs that bypassed spec, controllers with business logic, and an AI that would cheerfully start writing code before anyone had agreed on what it was supposed to do.

The tool wasn't the problem. The absence of structure around the tool was.

"The AI assistant was brilliant in a vacuum but inconsistent under pressure. We needed guardrails, not limitations."
The core insight that started this project

So I built a plugin. Not a simple prompt library — a proper Claude Code plugin with commands, skills, scoped rules, and a memory system. Something that would enforce the way we want to work, not just assist with how we happen to be working.

Six months and seventeen versions later, here's what I learned.

What ICEA actually means in practice

The centrepiece of the plugin is a workflow called ICEA — Intent, Context, Examples, Acceptance. It's not a document format. It's a gate.

When a developer types "build a search filter for the deals list," the plugin intercepts the request before any code is written. It drafts a complete ICEA document — extracting the feature from the message, identifying the affected stack layer from a pre-built domain map, filling in constraints from the architecture docs, proposing test scenarios from the intent.

Then it stops. Completely. And asks for approval.

No code gets written until the developer replies APPROVED. Not "looks good." Not "sure." The literal word APPROVED. Everything in the hard rules is explicit about this because I learned the hard way that "fine, proceed" is not the same as having actually read the spec.

The four ICEA sections — each serves a specific downstream consumer in the workflow

The key insight I had early: the ICEA is not documentation, it's a contract. The Acceptance Criteria section drives the test scenarios. The Context section seeds the PR description. The Intent maps to the ADO work item. Every downstream output in the workflow traces back to something in the ICEA.

Dream: the memory system I wish every AI had

The second major piece is the Dream memory system. Claude Code sessions are ephemeral. Every session starts cold. If you spent three hours figuring out that the authentication middleware needs to be ordered a specific way, the next session has no idea that happened.

Dream reads recent sessions, extracts knowledge entries — patterns that worked, approaches that failed, architectural decisions and their rationale — and consolidates them into memory/topic-*.md files and a promoted facts section in CLAUDE.md. Every proposed change comes with a confidence score and a justification. You review and approve before anything is written.

Dream runs every 5–8 sessions, consolidating knowledge the way the brain consolidates memories during sleep

The analogy is deliberate. Dream is named after REM sleep precisely because it does for AI sessions what REM does for human memory — prunes what's irrelevant, strengthens what's important, and surfaces patterns that weren't obvious in the moment.

The rollback command I added recently matters more than I expected. When Dream promotes something incorrect — it happens, maybe once a sprint — being able to reverse that specific run without manually hunting through CLAUDE.md is the difference between a trustworthy system and a fragile one.

The token efficiency problem no one warned me about

Three months in, I ran a token analysis and got an uncomfortable number. Every skill was treating every run as a cold start. The code review skill was re-scanning all 50 files daily even though 48 of them hadn't changed in a week. The security skill was loading Python, JavaScript, and Java reference material for a pure C# codebase. The ICEA skill was reading raw source files to orient itself — thousands of lines it would never use.

The file-cache.json primitive tracks character counts — unchanged files are skipped entirely on subsequent runs

The fix was architectural: a shared primitives layer. A domain-map.md that maps the codebase to entry-point files — so skills can orient in one read instead of scanning hundreds. A file-cache.json that tracks character counts per file — so code review and security skip everything unchanged since last run. Scope flags (--changed, --pr, --full) so developers can target a pre-PR check at just the files they touched.

The savings were real. Daily code review on a medium project dropped from scanning 50 files to scanning 2 or 3. The ICEA skill, which used to open half the codebase to orient itself, now reads a single markdown file.

The PR compliance gate: the change that actually changed behaviour

Of everything I've built, the PR creation gate has changed developer behaviour more than anything else. It works like this: when a developer asks to create a PR, the pr-create skill automatically runs icea-review — a compliance check against the approved ICEA — before showing the confirmation prompt.

If the review returns BLOCKED, the PR cannot be created. The developer sees exactly which Acceptance Criteria are missing and what needs to change. There is an override flag (--skip-icea-check) for genuine emergencies, but it gets noted in the PR description — creating an audit trail.

The thing that surprised me most

Developers started writing better ICEA documents — not because I told them to, but because a bad ICEA meant their PR would get blocked. The gate created the incentive to get the spec right at the start. Which is the whole point.

The compliance report maps every changed file to a specific Acceptance Criterion. If a file touches behaviour that isn't in any AC, that's scope creep — flagged explicitly. If an AC has no corresponding code change, that's a missing implementation — flagged as BLOCKED.

The same gate also checks all three finding ledgers — code-review, security, and dynamic-scan — for open Critical findings. The check is deliberately soft: the developer can acknowledge and proceed, but the acknowledgement is recorded in the PR description. A reviewer can see at a glance whether a PR was opened with known Critical defects outstanding, and ask why.

The critic: catching problems before they reach disk

For a long time the plugin had two quality gates: the spec (you approve the ICEA before code is generated) and the diff (code-review and the PR gate check what you committed). There was a blind spot between them. In the moment the AI generates code, nothing reviewed it — and by the time code-review runs, the approved spec has dropped out of context. Code-review can tell you the code is correct. It cannot easily tell you the code does exactly what the AC asked, and nothing more, because it no longer has the AC in front of it.

The critic fills that gap. It is a second-pass reviewer that runs while both the ICEA and the freshly generated code are in context at the same time — the one moment where intent and implementation can be compared directly. It fires automatically at two points inside the ICEA flow:

1
On the ICEA draft — before you approve. It checks the spec for completeness, testability, B1–B7 coverage, and scope creep against the stated Intent. Its concerns don't appear as a separate wall of text; they fold into the same gap list you already review before approving.
2
On the generated code — before a single file is written to disk. It checks ICEA traceability (does every artefact map to an approved AC, and did anything get generated that no AC asked for?), simplicity (is this over-engineered?), rules compliance, decision transparency, and hidden assumptions the generator made where the spec was silent.

The second gate is the one that changes behaviour. The verdict controls the write. A clean verdict saves the files and moves on. A REVISE verdict regenerates the affected code — addressing the specific concerns — and re-critiques, up to two automatic attempts. If it still isn't satisfied after two tries, it stops guessing and surfaces to the developer: accept the code as-is, give specific guidance and retry, or halt and write nothing. There's a diminishing-returns guard too — if a retry produces the same concerns with no movement, it surfaces early rather than burning the second attempt.

Why a skill, and why ephemeral

It had to be a skill rather than just a command, because its whole value is running automatically inside the generation flow — a command only fires when a developer remembers to type it. But we added a standalone /critic icea|code as well, so the same critique is available on demand for a single phase. And it deliberately keeps no ledger and assigns no fingerprints: persistent finding-tracking already belongs to code-review, security, and dynamic-scan. Duplicating that machinery would blur the critic's job. It is a gate, not a tracker — it decides whether an artefact is good enough to save, then gets out of the way.

The lesson about source file transparency

Six months in, I realised something uncomfortable: Claude was reading source files without telling me it was doing so. The ICEA skill would silently open a controller to orient itself. The PR spec review would read every changed file in full before I knew it had started.

The plugin now has a formal consent model with three categories:

Category A — scan tools (code-review, security-review): implicit consent when you invoke the command, but the skill announces what it will read and why before the first file opens.
Category B — review skills (icea-review, pr-spec-review): the skill asks before reading each source file — which file, why it cannot answer without it, what it is looking for, and how many tokens it will cost.
Category C — never reads source (session-start, dream-status, dream, icea-feature): hard rule, never reads source files regardless of what you ask.

The gate format for Category B looks like this:

📂 Source file access request

  File       : src/services/MatterService.cs
  Why        : Diff shows the method signature changed but not
               whether the null guard was added
  Looking for: filterMatters() null check on roleId parameter
  Token cost : ~800 tokens

The developer says yes, no, or suggests a different file. If they say no, the skill continues with noted uncertainty. This sounds like friction. In practice it is the opposite — you know exactly what Claude is doing and why at every step.

What I'd do differently

The three things I underestimated:

1
The cost of stateless skills. I built each skill as an independent unit. None of them shared state. The token costs that resulted were painful and avoidable. I'd establish the shared primitives layer on day one, not month three.
2
The importance of the .gitignore from the start. Having a PAT in .claude/settings.json that isn't gitignored is a real credential leak risk. Simple to solve, easy to forget. The plugin now handles this at setup — but I had a close call.
3
Orientation documents beat raw source reads. Every skill that reads source files for context should instead read architecture documents. The domain-map is five minutes of work. The source scan it replaces is five seconds of latency on every single invocation.

Why we made these design and architecture decisions

None of this was designed up front. Each major decision came from a specific failure or cost we hit in production — and each one solved a concrete problem. If you're building something similar, the why matters more than the what:

1
Shared primitives layer — because three skills meant three copies of every bug. Code review, security, and ICEA each had their own scan logic, their own assumptions, their own state. A fix in one never propagated to the others. We pulled the common pieces into skills/shared/ as a single source of truth, so a change to a scope flag or cache rule is made once and inherited everywhere. It also made adding the v1.10 dynamic-scan skill cheap — it reuses the existing flags, model routing, consent model, and severity overrides instead of reinventing them.
2
Cache-aware scanning — because we were paying full price every single day. The original skills rescanned all 50 files on every run, even when only two had changed. file-cache.json tracks a character count per file; the first run is a full baseline, and every run after that processes only what changed. Token cost now scales with the size of the change, not the size of the project — a 60–95% reduction per invocation after the baseline.
3
The domain-map — because the AI was re-reading the whole codebase to orient itself. ICEA used to open raw source files on every feature request just to figure out where things live. domain-map.md maps feature areas to entry-point files with a fingerprint for staleness detection, so a skill orients in one read instead of hundreds. Your senior engineers don't re-read the codebase every morning; the tooling shouldn't either.
4
The architect skill — because someone had to build and maintain that map, and doing it by hand didn't scale. The domain-map only pays off if it exists and stays current, and hand-authoring one for every repo was exactly the kind of tedious, error-prone work that gets skipped under deadline. The architect skill detects the repo type, populates the architecture docs, and generates the domain-map.md with its staleness fingerprint automatically — and refreshes it as the codebase moves. Adding it removed a whole class of pain: stale or missing maps, inconsistent architecture docs across projects, and the manual upkeep that no one wanted to own. It turned the domain-map from a nice idea into something that's actually there when every other skill reaches for it.
5
The ICEA approval gate — because rework was the real cost, not typing. A blocked PR is expensive; a thin spec is the cause. Forcing an approved Intent-Context-Examples-Acceptance document before any code is generated felt like overhead until we measured it: one well-written spec prevents roughly three PRs that would have bounced. The gate changed behaviour through incentive alignment, not enforcement — developers write better specs because the cost of a bad one is now visible up front.
6
The source-file consent model — because "trust me, I read some files" isn't good enough. Every skill now declares a category: A (announce what it will read, then scan), B (ask before each file, with the reason and token cost), or C (never reads source). It sounds like overhead; it turned out to be the most useful change in two versions, because you always know exactly what Claude is reading and why — and reviewers stopped second-guessing the tool.
7
Business-context severity (B1–B7) — because a CVSS score doesn't know your domain. A "medium" technical finding that exposes immigration IDs or privileged client data is not medium in a law firm. The B1–B7 overrides escalate findings to Critical based on what data is actually at risk, applied consistently across every review skill — static and, now, dynamic. The tool's risk rating reflects the business, not just the textbook.
8
Dream, and then dream-rollback — because memory without an undo is a liability. Dream consolidates lessons between sessions so Claude stops relearning the same conventions. But once it started writing to CLAUDE.md and memory files, a wrong promotion or an over-eager delete had no undo. Because the dream-log is append-only and records the before-state of every change, we could add /dream-rollback to reverse any run — and the rollback is itself logged, so it's auditable and reversible too.
9
dynamic-scan (DAST) — because static analysis can't see what only happens at runtime. The security skill reads source and catches a lot, but it structurally cannot confirm an authentication bypass, an IDOR, a missing security header, or a misconfigured endpoint on a running app. dynamic-scan drives OWASP ZAP through Docker to test the live application, defaulting to a safe passive baseline plus a dependency audit, and gating attack-payload scans behind an explicit "not production, and I'm authorised" confirmation. SAST tells you what the code says; DAST tells you what the deployed system actually does. We needed both.
10
The critic layer — because the gap between "generated" and "committed" had no reviewer. We already gated the spec (ICEA approval) and the committed diff (code-review, the PR gate). But nothing checked the code in the moment it was produced — and by the time code-review runs, the spec has fallen out of context, so it can verify that the code is correct but not that it does exactly what the approved AC asked, no more and no less. The critic closes that gap with a generator-critic pass that runs while the ICEA and the freshly generated code are both in context. It fires at two gates inside icea-feature: once on the ICEA draft (its concerns fold into the gap list you review before approving), and once on the generated code before any file is written to disk. On the code gate the verdict controls the write — a clean verdict saves the files, a REVISE verdict regenerates up to twice and then surfaces to the developer with accept / guide / halt. We made it a skill, not just a command, precisely because it has to run automatically inside the existing flow — but we added a standalone /critic icea|code too, so the same critique is available on demand for a single phase. Deliberately, it keeps no ledger and assigns no fingerprints: persistent tracking already belongs to code-review, security, and dynamic-scan. The critic is a gate, not a tracker — it decides whether an artefact is good enough to save, then gets out of the way.
11
Going language-agnostic — because the workflow was the value, not the stack. The plugin began life assuming one stack: .NET, Angular, Node.js. But the ICEA gate, the memory system, the consent model, the critic — none of those are language ideas. They're workflow ideas. The stack assumption was woven through five layers (rules, architecture detection, review checkers, dynamic-scan configs, and the code-generation order), and only one of those — the generation order — was truly hard to unpick, because it's the part the critic gates. The fix wasn't to add stacks; it was to stop hardcoding any. Detection routes by repo-type signal; generation and task-breakdown read the active layers from architecture.md; review and scanning dispatch to per-language references by file extension. Now the same loop runs on .NET, Java/Spring Boot, Python (FastAPI/Django/Flask), and Node.js without the workflow knowing or caring which. The guard against regression is a language-coverage matrix in the validator: a language isn't "supported" until it has a rule file, a review checker, and an architecture template — so a half-added language fails the build instead of silently misbehaving. Adding the next language is one matrix row plus the files it names.

The thread connecting all eleven: every one started as a cost or a risk we could measure, and the architecture was the answer to a specific question — what is this tool doing that it shouldn't have to, and what can't it see that it needs to?

Persistent finding ledgers: closing the review-to-fix loop

For most of the plugin's life, only code-review had a persistent ledger. Run code-review, get an HTML report with FP-fingerprint IDs, run /fix FP-xxxxxxxx, watch the remediation apply, ledger marks it Fixed. The loop worked.

Security and dynamic-scan, however, only produced HTML reports. A developer who found a Critical SQL injection in the security report had to fix it manually — no audit trail, no cross-run tracking, no way to know on the next scan whether they had actually closed it.

The architectural fix was to formalise a finding-producing skill contract: every skill that surfaces actionable defects must write a persistent ledger with FP-fingerprint IDs. Security now writes security/security-ledger.md. Dynamic-scan writes dynamic-scan/dynamic-scan-ledger.md. Both follow the same schema as code-review, with one addition: findings that have no source-level fix (server config, TLS settings, infrastructure-level issues) get a manual-fix-required status instead of an FP ID — they're tracked but excluded from the automated fix workflow.

/fix FP-xxxxxxxx now searches all three ledgers, locates the entry in whichever one contains it, applies the remediation, and moves the entry from ## Open Findings to ## Fixed Findings in that specific ledger. The Summary counts update automatically. The review-fix-verify loop that used to work only for code-review now works for security findings and DAST findings too.

This unlocked the downstream gates. checkin Step 5c (Check D) reads all three ledgers for open Critical/High findings and blocks commits when any are present. pr-create Step 3c shows open Critical counts in the PR confirmation. dream-status check 1o aggregates the counts as a health-dashboard metric. None of these were possible while two of the three skills were write-once HTML.

Why this matters more than it looks

An AI assistant that finds defects has limited value. An AI assistant that finds defects, tracks them across runs, attributes their resolution to specific commits, and refuses to let you ship until they're closed — that's a different category of tool. The ledger contract is what makes the difference.

What's in the plugin today

After seventeen versions the plugin includes 18 skills, 22 commands, 6 scoped rule files, and 8 shared governance specs. The highlights:

→
icea-feature: The ICEA gate — blocks code generation until the spec is approved. Reads architecture-deployment.md for hosting/auth context; shows a visible advisory banner if deployment context is missing so reviewers know which ACs are placeholders. Generates in whatever stack the repo actually is — it reads the active layers from architecture.md rather than assuming one.
→
architect: Detects the repo type and scaffolds the architecture docs and domain-map. Supports .NET, Java/Spring Boot, Python (FastAPI/Django/Flask), Node.js, Angular, and React — the detection result is what makes every downstream skill language-agnostic. A validator coverage matrix prevents a language from being half-added.
→
critic: A second-pass reviewer that runs inside the ICEA flow — once on the draft spec, once on generated code before it is written to disk. The code-gate verdict controls the write, regenerating up to twice before surfacing to the developer. Ephemeral by design: no ledger, no fingerprints. Also available standalone via /critic icea|code.
→
dream / dream-rollback: Memory consolidation with full audit trail and per-run rollback. Ledger files are explicitly excluded from rollback — they track real defects, not memory.
→
code-review: Static analysis with persistent ledger (CodeReviews/code-review-ledger.md) — every finding gets an FP-fingerprint ID and a git-attributed Fixed status across runs.
→
security: OWASP/CVSS/CWE review with its own persistent ledger (security/security-ledger.md). FP-fingerprinted findings for source-fixable issues; manual-fix-required status for advisory/infrastructure findings.
→
dynamic-scan: Runtime DAST with OWASP ZAP via Docker — the live counterpart to static security review, covering Angular / ASP.NET MVC / Web API / Blazor targets plus dependency auditing. Writes a third persistent ledger (dynamic-scan/dynamic-scan-ledger.md) with source-mapped fixes.
→
/fix: Multi-ledger remediation. Takes any FP-xxxxxxxx fingerprint, searches all three ledgers, shows the vulnerable code and fix side by side, waits for explicit confirmation, then applies with str_replace and moves the entry from Open Findings to Fixed Findings in its ledger. The review-fix-verify loop now works for code, security, and DAST findings.
→
checkin: Pre-commit gate — code quality (Check A) + ICEA compliance (Check B) + secrets scan (Check C) + open security/DAST findings (Check D) in one pass. A known Critical finding in any ledger blocks the commit unless explicitly overridden with --skip-security-gate and a written justification logged in the commit message.
→
pr-create: Auto-runs the icea-review gate before any PR is submitted. Step 3c also checks all three ledgers for open Critical findings and surfaces them in the confirmation prompt — the developer can acknowledge and proceed, but acknowledgement is noted in the PR description.
→
architect: Detects repo type, populates architecture docs, generates domain-map.md, and runs the deployment questionnaire — hosting model, auth strategy, environments, CI/CD — captured once in architecture-deployment.md and reused by icea-feature, app-readiness, and pr-create.
→
dream-init: One-time project setup. Deploys 22 command stubs, seeds caches, runs the architect questionnaire. Now also creates or updates .gitignore automatically — plugin entries always added, root-level artifacts auto-detected, full repo walk for bin/, obj/, .env etc. with a developer selection prompt. Declined entries are tracked so they aren't re-prompted.
→
dream-status: Read-only health check — green/amber/red on 15 infrastructure items including the new open findings (all ledgers) check that aggregates Critical / High / Medium counts across all three ledgers.
→
app-readiness / plugin-readiness: Production go-live assessments — 8-domain Enterprise Architect review of the application, 6-domain AI Architect review of the plugin itself. Both read deployment context and finding ledgers to give a maturity score per domain.
→
sprint-metrics: Post-sprint KPI report via ADO REST API — ICEA compliance rate, PR rejection rate, rework hours.
→
token-analysis: Session consumption analysis with skill usage telemetry — surfaces which skills are never used.
→
ado-tasks / icea-feature (as explicit commands): Both auto-trigger on natural-language keywords, but also have explicit /ado-tasks and /icea-feature slash commands for deliberate invocation — useful when the trigger doesn't fire or when re-generating an existing artifact.

If you're using Claude Code on a distributed team and you haven't built structure around it, you're getting half the value. The model is the easy part. The workflow is the hard part — and it's where most of the leverage is.
