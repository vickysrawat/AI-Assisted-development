Immediately after SAVE ICEA, Claude generates the full Tech Spec and writes it to temp/ADO-{ID}-tech.md. Open in VS Code preview (Ctrl+Shift+V) — same pattern as the ICEA review. The ICEA is already populated from the plan — no re-asking answered questions. Key sections:

ℹ Data access always uses Dapper with parameterised SQL. EF Core is never used.
⚖ Critic gate (Tech Spec). Before the draft is written to temp/, the auto-critic checks it against the approved ICEA — every AC maps to a planned file, no planned file exceeds the ACs, each selected D-option is reflected in the design, and the coverage matrix and test cases are complete. On a REVISE verdict it regenerates the Tech Spec (up to 2× automatically); a genuine ICEA fault is surfaced with a REVISE ADO-{ID} recommendation rather than being papered over. This is the cheapest point to catch a spec↔design gap — before any code is generated.

Q
Code review
A Coverity-style static analysis with persistent tracking, stable fingerprints, and one-command fixes.
A Run a scan /code-review ›

Runs a three-pass analysis: Pass 1 deterministic rule-based checkers, Pass 2 persona passes for reliability, concurrency, and API-contract, and Pass 3 a free-flow adversarial "what breaks next" sweep. It is stack-agnostic — it detects the language and loads only the matching checkers.

Scans are cache-aware via .claude/file-cache.json, so unchanged files are skipped on the default scan. Results are written as HTML + Markdown reports plus a running ledger at CodeReviews/code-review-ledger.md, where every finding gets a stable FP-xxxxxxxx fingerprint.

command
You: /code-review --changed

Passing no flag opens an interactive scope menu. Otherwise pick a scope flag:

FlagScope
--changedStaged + unstaged files only
--prBranch diff vs base
--fullAll files, ignore cache
--ciFull scan + cache warning if a cache file is found
--area backend|frontend|config|<Module>One area or knowledge-graph module
--continueResume from checkpoint
iNo flag opens an interactive scope menu. --full and --ci never consult the cache.
B Apply a fix /fix ›

Give /fix a fingerprint from the ledger and it locates the entry, applies the fix directly to source, and marks the finding Fixed — no re-scan needed.

command
You: /fix FP-1a2b3c4d
C Dismiss a false positive /dismiss ›

Suppress a finding with an auditable reason category and a required justification. The finding moves to the ledger's Dismissed section and is skipped on future scans.

command
You: /dismiss FP-1a2b3c4d false-positive reason text
iDismissals re-open automatically if the code at that location later changes, so a stale suppression can never hide a new defect.

Static SAST for OWASP/CWE/IaC plus dynamic DAST via OWASP ZAP, both with persistent FP-fingerprinted ledgers.
A Static security review /security-review --changed ›

SAST covering the OWASP Top 10, CWE weaknesses, and IaC / cloud-posture misconfiguration. It uses the same three-pass architecture and the same scope flags as code review, and writes security/security-ledger.md with false-positive fingerprints so dismissed findings stay quiet on later runs.

command
You: /security-review --changed
iBusiness Context Severity (B1–B7): findings that touch sensitive data — immigration IDs, privileged legal matter, vulnerable-client data — are escalated (e.g. to Critical) regardless of raw CVSS score.
FlagScope
--changedStaged / uncommitted files only
--prBranch diff vs target
--fullEntire working tree, ignore cache
--areabackend | frontend | config | <Module>
B Dynamic scan (passive baseline) /dynamic-scan ›

DAST via OWASP ZAP run in Docker. The default run is a passive baseline plus a dependency audit (npm / pip / NuGet) — safe against any target. Adding --full runs an active scan that sends attack payloads. Results land in dynamic-scan/dynamic-scan-ledger.md.

command
You: /dynamic-scan --url http://localhost:5000

Baseline strategy: the first run records all current findings as a baseline and never gates. Only new findings versus that baseline gate on later runs, so pre-existing noise never blocks a check-in.

iProduction gate: a hard environment gate refuses to run against production. Point --url at a local or non-prod environment only.

Pre-commit check
One unified quality gate that runs four checks and emits a single pass/fail verdict before you commit.
A Run the gate /checkin ›

Runs four checks in a single pass over your staged and modified files, then emits one verdict. When everything is green it hands back a pre-filled git commit command so you can commit immediately.

command
You: /checkin
CheckWhat it verifies
A Static analysisRuns code-review over staged/modified files for code-quality defects.
B ICEA complianceRuns icea-review over the diff — tier re-verification and manifest delta included.
C Secrets scanScans staged files for secrets and sensitive files; confirms .claude/settings.json is ignored.
D Findings gateBlocks on open Critical/High findings across the code-review, security, and dynamic-scan ledgers.
iThe same gates are also enforced non-interactively by the icea-floor and findings-gate-precommit hooks installed by /setup-init (as .sh, .ps1, or .cjs depending on the detected shell environment).
B Read the A/B/C/D verdict info ›

The output shows a one-line result for each of the four checks and a single overall verdict. When green, it appends a ready-to-run commit command; when blocked, it lists each blocking issue with a one-line fix and withholds the commit command.

iA green verdict means the same gates the pre-commit hooks enforce have already passed — the commit will go through cleanly.
iCheck C (secrets) and Check D (findings gate) always run and block on failure — resolve or dismiss open Critical/High findings before committing.

M
Memory & the Dream system
A two-layer capture pipeline feeds memory/MEMORY.md, then Dream consolidates it into durable, scored project intelligence.
A Two-layer capture auto ›

Capture is deliberately redundant so nothing valuable is lost:

Layer 1 — proactive instruction. A hard rule in CLAUDE.md tells Claude to write to memory/MEMORY.md the moment a trigger fires, in the same response, before replying to the next message.

Layer 2 — Stop hook safety net. .claude/hooks/memory-capture.sh injects a checklist reminder at the end of every turn, catching anything Layer 1 missed.

imemory/ is exempt from the Write Gate — memory entries are written directly, no APPROVE needed.
B The five capture triggers reference ›

When any of these fires, an entry is written immediately:

TriggerWhat to capture
Plan approvedApproach agreed, tools chosen, constraints set
Task completedPattern that worked, convention confirmed
Error resolvedError + root cause + fix + gotcha to avoid repeating
Approach abandonedWhat failed and why — so it is not retried
Architecture decisionDecision + rationale + alternatives rejected
C Consolidate & inspect /dream ›

/dream — run every 5–8 sessions. It scores entries by confidence, decay, and multiplicity, then proposes ADD / UPDATE / DELETE operations for tiered approval before writing anything. A 20-entry promotion cap keeps MEMORY.md lean; overflow is demoted to memory/topic-*.md.

command
You: /dream

/dream-health — generates memory/health.html, a dashboard of confidence scores, the decay curve, promote candidates, and run history.

command
You: /dream-health
D Audit & rollback /dream-audit ›

/dream-audit — a quarterly quality audit that flags uncited facts and contradictions.

/dream-rollback — reverses a specific consolidation run using the audit trail in memory/dream-log.md.

iThe capture and Stop hooks are wired automatically by /setup-init and refreshed by /setup-sync — no manual hook configuration required.
G
The knowledge graph
A typed, confidence-scored dependency graph of the codebase that stays fresh incrementally.
A How the graph is stored graph.json ›

.claude/graph/graph.json is authoritative. It holds typed nodes (module, service, library, external) and typed edges, each carrying a confidence level.

Edge confidenceMeaning
EXTRACTEDScript-derived from imports — deterministic.
INFERREDDerived from DI, config, or dynamic wiring.
AMBIGUOUSA candidate relationship that could not be resolved with confidence.

Each module carries a fingerprint — a hash over its source — so any file change marks that module stale. Human-readable markdown projections (graph-index.md breadth table plus per-module detail files) are generated from graph.json and are never hand-edited.

.claude/graph/ graph.json authoritative typed graph graph-index.md generated breadth table modules/ generated per-module detail graph.html /graph-viz output .stale flag written by the post-merge hook
B Refresh the graph incrementally /graph-sync ›

Recomputes fingerprints and regenerates only stale modules, then reconciles removed, renamed, and orphaned modules. Typed edges are re-derived — EXTRACTED edges are produced deterministically by scripts/graph-extract-edges.js. Past 30 modules a flat graph is restructured into domain groups. Finally it regenerates the markdown projections and deletes the .claude/graph/.stale flag on success.

command
You: /graph-sync
iA post-merge hook (graph-stale-detect.sh) writes the .stale flag when tracked entry-point files change, prompting you to run /graph-sync.
C Visualize the graph /graph-viz ›

Renders a self-contained, fully offline HTML visualization at .claude/graph/graph.html — 2D SVG by default; --3d uses a locally vendored WebGL library. Nodes are coloured by type, edges styled by type and confidence, hub ("god") nodes and stale modules are flagged, and hover shows a module's dependencies and dependents. It reads graph.json only.

command
You: /graph-viz --3d

Skill Responsibilities
Each skill owns exactly one phase. No skill crosses into another's territory.
SkillOwnsNever does
icea-feature Drafts Plan, ICEA, Tech Spec, Tracker, Epic doc. Handles EDIT/REJECT cycles. Never generates implementation code. Never writes ✅ Approved. Never handles revision inline.
icea-approve Reads ICEA from disk. Presents summary. Writes Status: ✅ Approved. Outputs ADO description block. Never generates implementation code.
icea-implement Reads approved ICEA + Tech Spec. Generates code. Runs auto-critic. Gates writes behind APPROVE ADO-{ID}. Updates tracker. Never writes ICEA, Tech Spec, or Tracker header. Never approves.
icea-revise Locates files by ADO ID. Confirms path. Checks for mid-implementation files. Applies edits. Re-gates only if was ✅ Approved. Never generates implementation code.
icea-status Reads all files for an ADO ID. Reports state and next action. Never writes anything — completely read-only.
N
Model routing
Skills pick a model by task type via environment variables, overridable per project.
A How routing works reference ›

Each skill routes to a model based on the kind of work it does. The mapping is driven by environment variables, so you can point generation, review, and infrastructure tasks at different models without editing any skill. Override any variable per project in .claude/settings.json under the env key.

iThe authoritative table of variables, defaults, and per-skill assignments lives in skills/shared/model-routing-spec.md.
B Routing variables reference ›

Set any of these in .claude/settings.json → env to route a task class to a specific model.

VariableRoutes
ICEA_MODELICEA, plan, tech-spec, and code generation
REVIEW_MODELCode review, security review, dynamic scan, PR/ICEA review
CRITIC_MODELThe critic pass — falls back to REVIEW_MODEL if unset
INFRA_MODELInfrastructure skills (checkin, dismiss, setup)
