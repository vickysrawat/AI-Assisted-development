01
Start a Feature
Describe the feature to Claude. The ICEA gate triggers and collects the identifiers it needs before drafting anything.
1.1 Describe your feature natural language ›

Just describe what you want to build. The gate triggers automatically on keywords like build, implement, add, create, I need, work on, ADO #. No special syntax required.

Claude Code — chat 
You: Add an export to CSV button on the report page so users can download the current view

Or invoke explicitly with all three identifiers upfront:

explicit invocation 
You: /icea-feature ADO-1847 R3 S12 — add export to CSV on the report page
1.2 Provide ADO, Release, and Sprint numbers required ›

If any of the three identifiers are missing, Claude asks for them all in one prompt. Never asked one at a time.

Claude: Before I create the ICEA, I need a few details: ADO #: [e.g. ADO #1847] ← the work item ID Release #: [e.g. Release 3 / R3] ← the release this ships in Sprint #: [e.g. Sprint 12 / S12] ← the sprint it is planned for
your reply 
You: ADO #1847, Release 3, Sprint 12
ℹ If an approved ICEA already exists for this ADO ID, Claude redirects you to REVISE ADO-1847 rather than overwriting it.
02
Review the Plan
Claude drafts a structured feature plan before the ICEA — problem statement, story, personas, MoSCoW, risks, assumptions. Review it interactively, correct wrong assumptions, then save.
2.1 Plan presented inline automatic ›

Before drafting the ICEA, Claude drafts a structured feature plan covering: problem statement, story (As a/I want/so that), personas, MoSCoW feature priority, release plan, assumptions, risks, pre-mortem, and open questions.

📋 FEATURE PLAN — ADO #1847 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Problem Statement: Users cannot export filtered report data — they manually copy rows. Success: export available to 80% of report users within Sprint 12. Story: As a Report Viewer, I want to export the current filtered view to CSV, so that I can analyse the data in Excel without manual copying. Personas: Report Viewer: analyst · runs weekly reports · needs offline analysis Must Have: Export filtered grid to CSV Won't Have: Excel format, scheduled exports, email delivery Assumptions: [1] Export is synchronous (small datasets) — unverified [2] CSV format only — verified Pre-mortem: "This shipped and failed — export silently truncated large datasets" ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Review above. Correct anything wrong or: SAVE PLAN ADO-1847
2.2 Correct the plan conversationally optional ›

Correct wrong assumptions or fill in unknowns before the ICEA is drafted. No special syntax — just reply naturally.

examples 
You: [1] Export can be large — use async with a download link
You: [3] Auth policy is RequireReportViewer not RequireMatter
You: [5] Hidden columns should not be included

Claude updates the plan after each correction. When the plan looks right, save the ICEA.

03
Review & Save ICEA
Claude drafts the ICEA from the confirmed plan. Review it, answer any remaining gaps, and save when ready. Nothing is written to disk until you say so.
3.1 ICEA drafted to temp file automatic ›

Claude generates the full ICEA from the confirmed plan and writes it to temp/ADO-{ID}-icea.md. Open it in VS Code for a clean read with properly rendered tables and headings.

ℹ Open with Ctrl+Shift+V to open VS Code markdown preview in a side panel. Keep chat on the left, preview on the right.

Claude lists any remaining gaps in chat — answer them there. The temp file is rewritten after each change so the preview stays current.

📄 ICEA draft written to temp/ADO-1847-icea.md Open in VS Code preview: Ctrl+Shift+V ⚠ ICEA GAPS — review in the file before saving: [1] Intent — Max file size: no limit specified [2] Context — Error state: what if the export service times out? Answer gaps here in chat. I will update the temp file after each change. When satisfied: SAVE ICEA ADO-1847
3.2 Answer gaps and make edits optional ›

Answer gaps by number or make freeform edits in chat. After each change Claude rewrites temp/ADO-{ID}-icea.md in place — VS Code preview auto-refreshes. Claude confirms with one line in chat.

examples 
You: [1] Max file size is 50MB
You: Change the endpoint to POST /api/reports/{id}/export
You: Add AC — export must complete within 30 seconds
You: Skip [2] — will decide later
✅ Updated — AC-F3 endpoint. Refresh preview.
3.3 Save the ICEA SAVE ICEA ›

When satisfied, save the ICEA. This writes the plan and ICEA to disk and immediately begins drafting the Tech Spec.

command 
You: SAVE ICEA ADO-1847
✅ ICEA saved — ADO #1847 Path: docs/Release3/Sprint12/UserStory1847/ADO-1847-export-to-csv.icea.md Temp file cleaned up. Drafting Tech Spec now...
✓ Plan is written at SAVE PLAN. ICEA is written at SAVE ICEA. Nothing else is written until SAVE TECH.
04
Review & Save Tech Spec
Claude drafts the Tech Spec from the saved ICEA. Covers architecture, data design, API contract, test cases, and sizing. Review it interactively before saving.
4.1 Tech Spec drafted automatically automatic ›

Immediately after SAVE ICEA, Claude generates the full Tech Spec and writes it to temp/ADO-{ID}-tech.md. Open in VS Code preview (Ctrl+Shift+V) — same pattern as the ICEA review. The ICEA is already populated from the plan — no re-asking answered questions. Key sections:

SectionContent
2. Architecture Touch PointsLayer, component/file, change type
3. Data DesignEntities, DTOs, Dapper repositories, SQL
4. API ContractEndpoint, auth policy, request/response, error codes
9. Testing StrategyPer AC — positive + negative test cases, SP estimate
11. Sizing SummaryTotal SP, Story vs Epic recommendation, story breakdown
ℹ Data access always uses Dapper with parameterised SQL. EF Core is never used.
4.2 Story vs Epic recommendation automatic ›

Section 11 totals the SP across all ACs and makes a recommendation.

STORY (≤ 5 SP)
Total SP: 5 Recommendation: STORY — fits in one sprint
EPIC (> 5 SP)
Total SP: 10 Recommendation: EPIC — exceeds single-story capacity Suggested story breakdown: Story 1 — Core export (AC-F1, AC-F2) — 5 SP — Sprint 12 Story 2 — Filters + async (AC-F3, AC-F4) — 5 SP — Sprint 13 Dependency: Story 2 → Story 1
4.3 Answer open questions and edit optional ›

Open questions are shown after the Tech Spec. Answer them conversationally or skip for later. Claude re-outputs only the changed section — not the full Tech Spec — after each change, so the conversation stays focused.

examples 
You: ❓[1] use a stored procedure for the export query
You: Section 4 — the endpoint returns a pre-signed URL, not a stream
You: Skip ❓[2] — will decide during implementation
4.4 Save the Tech Spec SAVE TECH ›

Save the Tech Spec when satisfied. Tracker and Epic doc are written automatically — no further input needed.

command 
You: SAVE TECH ADO-1847

If open questions remain, Claude warns before saving:

⚠ 1 open question remains in the Tech Spec. Will be saved as ❓ placeholder — resolve before APPROVE ADO-1847. ❓ [2] API Contract — stream or pre-signed URL? Reply SAVE TECH ADO-1847 CONFIRM to save anyway.
save with open questions 
You: SAVE TECH ADO-1847 CONFIRM
✅ Documents saved — ADO #1847 · Release 3 · Sprint 12 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Plan → ...ADO-1847-export-to-csv.plan.md ICEA → ...ADO-1847-export-to-csv.icea.md TechSpec → ...ADO-1847-export-to-csv.techspec.md Tracker → ...ADO-1847-export-to-csv.tracker.md Share with your Tech Lead and Product team for review. When approved: APPROVE ADO-1847
05
Share for Review
Share the saved documents with your Tech Lead and Product team. Check status at any time — even from a new session.
5.1 Check status at any time STATUS ›
command 
You: STATUS ADO-1847
📊 STATUS — ADO #1847 · Export to CSV ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Type: STORY | Release: 3 | Sprint: 12 Files: ✅ Plan — saved ✅ ICEA — Status: DRAFT ✅ TechSpec — Open questions: 1 ✅ Tracker — 0/4 ACs done Next action: → Share with Tech Lead, then APPROVE ADO-1847
5.2 Revise based on feedback REVISE ›

If the Tech Lead or Product team requests changes, revise the documents before approving.

command 
You: REVISE ADO-1847

Claude locates the files, confirms the path, shows current state and open questions, and asks what needs updating. All edits are conversational — files re-written once changes are confirmed.

ℹ Status-aware re-gate: if the ICEA is still DRAFT, revising does not change the status — the gate is already active. Only if the ICEA was ✅ Approved does revising reset it to DRAFT — Revising and re-block code generation until re-approved.
⚠ If you revise the ICEA and it significantly changes scope, Claude will warn that the Tech Spec may be out of sync and suggest running TECH ADO-1847 to re-draft it.
06
Approve
Once the Tech Lead and Product team are satisfied, approve the ICEA. Works from any session — Claude reads state from disk.
6.1 Approve the ICEA APPROVE ›

Run from any session — even a brand new one. Claude reads the ICEA from disk, shows a lightweight summary, and writes the approval status.

command 
You: APPROVE ADO-1847
📋 ICEA APPROVAL — ADO #1847 · Release 3 · Sprint 12 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ICEA: ...ADO-1847-export-to-csv.icea.md — Status: DRAFT Intent: Export report data to CSV with column selection Type: STORY | Total SP: 5 Open questions: 1 ← consider resolving before approving Reply APPROVE ADO-1847 to confirm.
confirm 
You: APPROVE ADO-1847
✅ ICEA Approved — ADO #1847 Status: ✅ Approved written to ICEA file To start implementation: IMPLEMENT ADO-1847
07
Implement
Claude generates all implementation code from the approved ICEA and Tech Spec. Code is reviewed by the auto-critic before you decide what to write to disk.
ℹ
Skill responsibilities: each skill has one job and never crosses into another's territory.
  • icea-feature — drafts Plan, ICEA, Tech Spec, Tracker. Never generates code.
  • icea-approve — writes ✅ Approved status. Never generates code.
  • icea-implement — generates and writes code only. Never writes ICEA or Tech Spec.
  • icea-revise — revises ICEA and Tech Spec only. Never writes code.
  • icea-status — reads and reports. Never writes anything.
7.1 Start implementation IMPLEMENT ›
story 
You: IMPLEMENT ADO-1847
epic — story by story 
You: IMPLEMENT ADO-1847 Story-1
... later ...
You: IMPLEMENT ADO-1847 Story-2

Claude checks the tracker for already-implemented ACs and skips them. Shows an implementation plan before generating any code.

7.2 Auto-critic runs before any disk write automatic ›

All code is generated in context first. The auto-critic checks for ICEA traceability, rules compliance, and hidden assumptions. If the critic fails, Claude revises automatically (up to 2 retries) before surfacing to you.

7.3 Write the implementation files APPROVE ›

After the critic passes, Claude lists all files to be written and waits for your approval before touching the filesystem.

📁 WRITE PENDING — reply APPROVE ADO-1847 to write all files ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ [1] src/Api/Controllers/ReportExportController.cs [2] src/Application/Services/ReportExportService.cs [3] src/Infrastructure/Repositories/ReportExportRepository.cs [4] src/Application/DTOs/ExportRequestDto.cs [5] tests/Api/ReportExportControllerTests.cs [6] tests/Application/ReportExportServiceTests.cs
write files 
You: APPROVE ADO-1847
✅ Implementation written — ADO #1847 AC-F1: ✅ Done AC-F2: ✅ Done AC-F3: ✅ Done AC-F4: ✅ Done Tracker updated. All ACs complete. ICEA marked COMPLETE.
08
Track & Bug Log
Log bugs as they are found during testing. The tracker is updated instantly — no approval needed.
8.1 Log a bug BUG ›
command 
You: BUG ADO-1847 — CSV includes hidden columns when filter is active
🐛 Bug logged — ADO #1847 Added to tracker: ...ADO-1847-export-to-csv.tracker.md
8.2 Review open bugs and tracker STATUS ›
command 
You: STATUS ADO-1847

Shows AC progress, open bugs, and fixed bugs. The tracker file can also be opened directly in any markdown viewer.

8.3 Mark a bug as fixed REVISE ›

Once a bug is fixed and verified, update the tracker via REVISE to record the fix notes and mark it ✔ Fixed.

command 
You: REVISE ADO-1847
then describe the fix 
You: Mark bug 1 as fixed — hidden columns now excluded via columnDef.hide check in ExportService
✔ Bug #1 marked fixed — ADO #1847 Tracker updated: ...ADO-1847-export-to-csv.tracker.md Fix noted: hidden columns excluded via columnDef.hide check
✓ Tracker updates (bug fixes, status changes) are written immediately — no APPROVE ADO-{ID} needed. Only source code requires approval before writing.

All Keywords
Every keyword the plugin recognises. No slash prefix needed — just type them in any session.
/icea-feature ADO-{ID} R{N} S{N}

Start a new feature. Triggers automatically on feature keywords too.

SAVE PLAN ADO-{ID}

Save plan to disk. Triggers ICEA draft automatically.

SAVE ICEA ADO-{ID}

Save Plan + ICEA to disk. Triggers Tech Spec draft automatically.

SAVE TECH ADO-{ID}

Save Tech Spec, Tracker, and Epic doc. Warns if open questions remain.

SAVE TECH ADO-{ID} CONFIRM

Save Tech Spec with open questions — bypasses the warning.

TECH ADO-{ID}

Draft Tech Spec from saved ICEA. Cross-session recovery.

ICEA ADO-{ID}

Generate ICEA from a saved plan. Cross-session recovery when plan is on disk but ICEA is not.

APPROVE ADO-{ID}

Approve ICEA after Tech Lead + Product review. Works in any session.

IMPLEMENT ADO-{ID}

Generate and write implementation code for an approved ICEA.

IMPLEMENT ADO-{ID} Story-{N}

Implement a specific story within an Epic.

REVISE ADO-{ID}

Revise ICEA and/or Tech Spec after feedback.

STATUS ADO-{ID}

Check current state and next action. Re-entry point after any session gap.

BUG ADO-{ID} — {description}

Log a bug to the tracker. Written immediately, no approval needed.

/skip-icea

Emergency override — skip the ICEA gate. Warns once then proceeds.

Skill Responsibilities
Each skill owns exactly one phase. No skill crosses into another's territory.
SkillOwnsNever does
icea-feature Drafts Plan, ICEA, Tech Spec, Tracker, Epic doc. Handles EDIT/REJECT cycles. Never generates implementation code. Never writes ✅ Approved. Never handles revision inline.
icea-approve Reads ICEA from disk. Presents summary. Writes Status: ✅ Approved. Outputs ADO description block. Never generates implementation code.
icea-implement Reads approved ICEA + Tech Spec. Generates code. Runs auto-critic. Gates writes behind APPROVE ADO-{ID}. Updates tracker. Never writes ICEA, Tech Spec, or Tracker header. Never approves.
icea-revise Locates files by ADO ID. Confirms path. Checks for mid-implementation files. Applies edits. Re-gates only if was ✅ Approved. Never generates implementation code.
icea-status Reads all files for an ADO ID. Reports state and next action. Never writes anything — completely read-only.
