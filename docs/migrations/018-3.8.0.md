# Migration — v3.8.0: Plugin lifecycle commands renamed dream-* → setup-*

## What changed

The four plugin lifecycle commands have been renamed to separate them from the Dream memory feature:

| Old name | New name |
|---|---|
| `/dream-init` | `/setup-init` |
| `/dream-sync` | `/setup-sync` |
| `/dream-teardown` | `/setup-teardown` |
| `/dream-status` | `/setup-status` |

The Dream memory feature commands are unchanged: `/dream`, `/dream-health`, `/dream-audit`, `/dream-rollback`.

## Action required

Run `/setup-sync`. It automatically:
- Removes the old `dream-init.md`, `dream-sync.md`, `dream-teardown.md`, `dream-status.md` stubs from `.claude/commands/`
- Deploys the new `setup-init.md`, `setup-sync.md`, `setup-teardown.md`, `setup-status.md` stubs

No manual steps needed.

## Internals renamed too (no action required)

The rename was completed through to the machinery, so nothing keeps the old name behind the
new commands:

- Bootstrap scripts renamed: `scripts/dream-init-bootstrap.cjs` → `scripts/setup-init-bootstrap.cjs`,
  `scripts/dream-teardown.cjs` → `scripts/setup-teardown.cjs`. All callers (command definitions,
  skills, the architect post-detect step) point at the new names.
- Test scenario fixtures renamed `dream-{init,status,sync}.yaml` → `setup-*.yaml`; validators
  (`tests/validate.js`, `tests/validate.py`) updated to open the renamed skills/scripts.
- Living docs (`CLAUDE.md`, `DEVELOPER-GUIDE.md`, `README.md`), command definitions, deployed
  hook comments, and shared specs updated to the new command names.

**Intentionally retained:** the state file `.claude/dream-init-state.json` and its keys
(`dream_init_plugin_version`, `dream_init_last_run`) keep their names. They are internal
identifiers read by `setup-status`/`setup-sync` in every already-provisioned project; renaming
them would make those projects read as unprovisioned (broken version-drift detection) with no
user-visible benefit. Historical records (CHANGELOG, WHITEPAPER, ADR prose, dated plan docs)
also keep their original wording.

## Also in v3.8.0 — Architecture doc set expands from 4 to 8 (ADR 0050)

The `architect` skill now generates four additional docs per stack —
`architecture-data.md`, `architecture-integrations.md`, `architecture-security.md`,
`architecture-decisions.md` — plus a Non-Functional Requirements section in
`architecture-deployment.md` and two Mermaid diagrams (End-to-End + Layered) in
`architecture.md`.

### Action required (existing projects)

New projects get the full 8-doc set at `/setup-init`. **Existing projects** already have the
original 4 docs; to add and populate the new ones:

- Re-run the architect skill ("populate architecture docs") — it copies the four new templates
  and populates only the missing files (existing populated docs are skipped), **or**
- Populate them individually as needed:
  - `/update-arch --data`
  - `/update-arch --integrations`
  - `/update-arch --security`
  - `/update-arch --deployment` (adds the NFR section)
  - `/update-arch --decisions` (start the decision log)

The two Mermaid diagrams appear the next time `architecture.md` is refreshed (`/update-arch`).
No content you have edited is overwritten; `architecture-decisions.md` is append-only.

## Also in v3.8.0 — Architect templates deduplicated (ADR 0051)

Internal-only refactor: the per-stack templates now live in a shared base
(`skills/architect/templates/_shared/`) plus per-stack overrides, composed at deploy time by the
bootstrap. **No project-side action required** — the composed output is byte-identical to the
previous per-stack templates, and `/setup-sync`'s skip-if-exists guard leaves every populated
`.claude/architecture/*.md` untouched.
