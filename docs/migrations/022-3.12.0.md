# Migration to 3.12.0
Summary: Share non-secret plugin config and durable artifacts, guarded against committed secrets. `.claude/settings.json` (team config), `.claude/architecture/`, and the three review ledgers are now **committed**; a new `check-settings-secrets.cjs` guard blocks any secret from reaching the shared `settings.json` at write-time (PreToolUse) and commit-time (pre-commit). Secrets and machine-specific permissions move to the gitignored `.claude/settings.local.json`.

## Sync actions (idempotent — never overwrite developer content)
Run `/setup-sync` (bootstrap `--mode sync`, which re-runs every `isDone`-guarded step):

- **New hook deployed:** `check-settings-secrets.cjs` is copied to `.claude/hooks/` and added to
  `.claude/hooks/.hashes` (setup-status check 1p verifies it). The `findings-gate-precommit.sh`
  pre-commit hook now also calls it with `--staged`.
- **PreToolUse wiring:** `stepWireSettings` adds a second `Write|Edit` PreToolUse entry —
  `node .claude/hooks/check-settings-secrets.cjs --hook` — alongside `icea-floor.sh` (only if not
  already wired).
- **Ignore block rewritten:** the VCS-aware ignore writer regenerates the managed block from the
  new `GITIGNORE_BASE`. Net effect on `.gitignore` / `.tfignore`:
  - **No longer ignored (now shared/committed):** `.claude/settings.json`, `.claude/architecture/`.
  - **Ledger negations added:** `CodeReviews/*` + `!CodeReviews/code-review-ledger.md`,
    `security/*` + `!security/security-ledger.md`,
    `dynamic-scan/*` + `!dynamic-scan/dynamic-scan-ledger.md`.
  - **Still ignored:** `.claude/settings.local.json`, checkpoints/caches, all dated reports, all
    `dynamic-scan/` `*.session`/`*.context` (plaintext-credential risk), `token-analysis/`,
    `prod-readiness/`, `temp/`.
- **setup-status check 1i inverted:** RED is now (a) a secret found inside the committed
  `settings.json`, or (b) the secret store `settings.local.json` not ignored — instead of the old
  "settings.json not ignored".

## ⚠ Manual step for projects that stored a PAT in `.claude/settings.json`
Un-ignoring `settings.json` means it becomes committable. If you used **Option B** (PAT in
`.claude/settings.json` → `env`) or have machine-specific `permissions` there:

1. Move the secret `env` (e.g. `AZURE_DEVOPS_PAT`) **and** the `permissions` block into
   `.claude/settings.local.json` (gitignored). Claude Code merges `settings.local.json` **over**
   `settings.json` at runtime, so behaviour is unchanged.
2. Keep only secret-free config in `settings.json` (hooks, `customInstructions`, non-secret `env`
   such as model routing `ICEA_MODEL`/`REVIEW_MODEL`).

The guards are the safety net if you forget: a `git commit` that stages a secret in
`settings.json` is **blocked** by the pre-commit hook (override for an emergency with
`SKIP_FINDINGS_GATE=1` and a written justification), and Claude is blocked from writing one via
the PreToolUse hook. `setup-status` (check 1i) reports the condition as ❌ Red.

## Notes
Claude Code deep-merges settings with `.claude/settings.local.json` overriding
`.claude/settings.json`. The plugin previously ignored **both** files out of caution ("PAT
credential storage"), which also trapped the non-secret team config the plugin needs. This release
adopts the intended split — shared file committed, local/secret file ignored — and makes
"no secrets in the shared file" a mechanical guarantee rather than a convention.

The detector flags two signals: secret-shaped **keys** (`*PAT*`, `*TOKEN*`, `*SECRET*`,
`*PASSWORD*`, `*API[_-]?KEY*`, `*CREDENTIAL*`, `*CONNECTIONSTRING*`, …) holding a non-placeholder
value, and secret-shaped **values** (Azure DevOps PAT, `ghp_`/`github_pat_`, AWS `AKIA…`, Slack
`xox…`, JWT, PEM private-key blocks). Placeholders (`your-pat-here`, empty) and env references
(`${AZURE_DEVOPS_PAT}`) are allowed, as are non-secret env keys like `*_MODEL`.

No data migration. Existing generated reports left in place (fix-forward); they simply stay
ignored. The three ledgers, once present, become tracked on the next `git add`.
