# Migration to 3.10.0
Summary: Architecture-doc "populated?" detection is now two-signal (retained TEMPLATE marker + scaffold-only body tokens); bootstrap no longer strips the marker (ADR 0053).

## Sync actions (idempotent — never overwrite developer content)
- Register the new shared spec `skills/shared/arch-populated-detect.md` (both `architect` and
  `setup-status` reference it). No project-side file is generated from it — it is skill-internal.
- No hook, ignore-block, rule, or state-file changes.

## Notes
The plugin code change is self-contained. There is **no data migration**: a project provisioned
under an older bootstrap has architecture docs whose `<!-- TEMPLATE -->` marker was stripped at
deploy time but whose bodies are still empty scaffolding. The new secondary detector keys on
population-proof scaffold tokens still present in those bodies (e.g. `[trust-zone diagram or list]`,
`<!-- From code:`), so:

- `setup-status` reports those architecture docs as **Amber — needs population** (was a false Green).
- Re-running the **architect** skill ("populate architecture docs") now correctly detects them as
  `NEEDS_POPULATION` and fills them, instead of skipping Steps 3–6.

Genuinely populated docs are unaffected — `⚠ Could not determine — needs manual input` is
deliberately **not** treated as a scaffold token (a populated file may legitimately contain it).
