# Migration to 3.6.0
Summary: Rule file system expanded from 8 to 43 files with frontmatter-based deployment.
Three rule files renamed. `directoryCatalog` added to `graph.json`. ADRs 0042–0045.

## Sync actions (idempotent — never overwrite developer content)

### Rule file renames (dream-sync re-deploys from plugin)
Three existing rule files are replaced by renamed equivalents. On `/dream-sync`, the old
files are removed from `.claude/rules/` and the new files are deployed:

| Removed | Replaced by |
|---|---|
| `.claude/rules/dotnet-rules.md` | `.claude/rules/csharp-dotnet-rules.md` |
| `.claude/rules/dotnet-framework-rules.md` | `.claude/rules/csharp-framework48-rules.md` |
| `.claude/rules/nodejs-rules.md` | `.claude/rules/nodejs-typescript-rules.md` |

### New rule files (deployed if stack detected)
Up to 35 new rule files may be deployed depending on the project's detected stack and
npm dependencies. They are selected via the frontmatter `detect:` discovery loop.
Existing `.claude/rules/` files are never overwritten — only absent files are added.

### `deployed_rules[]` added to dream-init-state.json
`session-start` now reads `deployed_rules[]` from `dream-init-state.json` instead of a
hardcoded `STACK_RULES` map. For projects upgraded from 3.5.0:
- If `deployed_rules` is absent, `session-start` falls back to the legacy `STACK_RULES`
  map (backward-compatible).
- Run `/dream-init` to regenerate `deployed_rules[]` with the full 43-file set.

### `directoryCatalog` in graph.json
The next `/graph-sync` or `/architect` run adds `directoryCatalog` to
`.claude/graph/graph.json` (optional top-level key — no breaking change). Skills that
read it fall back to a runtime `find` if the key is absent, so no action required before
the next graph refresh.

**If `directoryCatalog.reviewed` is already `true`** (set by a pre-3.6.0 security review
that lacked config-based scanning), dream-sync resets it to `false` so the developer is
prompted to re-validate the complete list on the next `/security-review`. This is a
one-time re-confirmation; subsequent scans skip the prompt as before.

```bash
node -e "
const fs = require('fs');
const p = '.claude/graph/graph.json';
if (!fs.existsSync(p)) process.exit(0);
const g = JSON.parse(fs.readFileSync(p, 'utf8'));
if (g.directoryCatalog && g.directoryCatalog.reviewed === true) {
  g.directoryCatalog.reviewed = false;
  fs.writeFileSync(p, JSON.stringify(g, null, 2) + '\n');
  console.log('  ↺ directoryCatalog.reviewed reset — re-validation required');
}
" 2>/dev/null
```

### `detect:` frontmatter on existing rule files
All 5 kept rule files (`project-rules.md`, `angular-rules.md`, `python-rules.md`,
`javascript-rules.md`, `java-rules.md`) now have `detect:` blocks. This is metadata only
and has no effect on how the files behave once deployed — no action required.

## Notes
- `/dream-sync` handles the file renames and new deployments automatically.
- No new hooks. No state file schema changes beyond `deployed_rules[]`.
- `plugin.json` `components.rules` updated from 8 to 43 entries — consumed by dream-sync
  to know which files belong to the plugin.
- `tests/validate.py` check 9 continues to hold for the graph — `directoryCatalog` is an
  additive key and does not affect node/edge validation.
