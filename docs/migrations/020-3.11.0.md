# Migration to 3.11.0
Summary: Disable Claude Code built-in auto-memory (`autoMemoryEnabled: false`) so Dream captures land in the repo `memory/` instead of the per-machine profile.

## Sync actions (idempotent — never overwrite developer content)
- Bootstrap sync mode (`setup-init-bootstrap.cjs --mode sync`, run by setup-sync Step 2) adds
  `"autoMemoryEnabled": false` to `.claude/settings.json` **only if the key is unset** — an
  explicit developer value (`true` or `false`) is preserved. No hook, ignore-block, rule, or
  state-file changes.

## Notes
Claude Code ships a built-in "auto memory" feature that is **on by default**. It injects a
per-turn instruction pointing at `~/.claude/projects/<project-slug>/memory/` and writes memory
in a one-file-per-fact + frontmatter format. That instruction competes with — and out-competes —
Dream's own capture path (the `memory-capture.sh` UserPromptSubmit hook + the CLAUDE.md `# Dream`
section), both of which target the **repo-relative** `memory/MEMORY.md`. The result was Dream
triggers writing to the machine profile (never committed, not shared, not seen by the dream
cycle) instead of the repo.

Setting `autoMemoryEnabled: false` removes the competing target so there is a single memory
system — Dream's repo-relative `memory/`. There is **no data migration**: entries already
captured under `~/.claude/projects/<slug>/memory/` are left in place (fix-forward). If any are
worth keeping, copy them into the repo `memory/MEMORY.md` by hand.

Reference: Claude Code memory docs (`code.claude.com/docs/en/memory.md`) — settings key
`autoMemoryEnabled`, env override `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`.
