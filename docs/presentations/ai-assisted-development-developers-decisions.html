The gate & its enforcement

Five decisions that turn "usually enforced" into "enforced".

The gate that blocks your own PRADR 0001

Problem

AI used inconsistently; rework dominated — most blocked PRs traced to a thin or absent spec.

Decision

No code until an ICEA is on disk and approved. "The ICEA is a contract, not documentation."

Lesson

Incentive changes behavior — a bad ICEA blocks your own PR, not someone else's.

Rejected: prompt-for-spec (non-deterministic) · review-then-code (spec falls out of context) · optional ICEA.

One good spec ≈ 3 PRs that don't bounce

Gate the output, not the inputADR 0002

Problem

Skills self-detected triggers — "load me when you see these words." The model implemented before the skill loaded, then self-corrected.

Decision

Hard rules fire on what the model is about to produce, not on what the user typed. Triggers are routing, not enforcement.

Lesson

"Enforcement that depends on the enforcer being consulted first will be bypassed."

Rejected: input-triggered rules · trigger + backup rule · hope for good behavior.
✗ Input trigger — "on these words, load me"
model already writing before the rule fires
→
✓ Output gate — checks the artifact
blocks the write, no matter what was typed

Constrain the artifact, not the conversation.

The gate that was silently offADR 0027

Problem

Re-running overwrote prior approvals (data loss). Investigating revealed: filename glob + emoji-status regex mismatch meant approved ICEAs failed the gate silently.

Decision

Re-run = revise-by-default (keyed by ADO ID); REVISE rewrites Status: → DRAFT to re-arm the gate; predicates fixed.

Lesson

The enforcement floor had a critical defect and no one knew. Test the gate, not just the happy path.

Rejected: "just warn on overwrite" · session-only state · silent data loss.

filename glob mismatch → no files matched
emoji in Status: → regex missed it

Both bugs meant the gate wasn't firing at all.

Approval that works across SlackADR 0031

Problem

Approve/implement assumed a live session; APPROVE after reopening did nothing — state lived in chat.

Decision

The ICEA Status: line is the single source of workflow state; skills read it from disk every invocation.

Lesson

Approval from email or Teams is actionable with one phrase. Session continuity never required.

Rejected: session-only state · dual truth chat+file (one always drifts).

Structure & orientation

How we stopped every run from starting cold.

One bug, three copies, unfixed foreverADR 0003

Problem

Three skills, three copies of every scan bug; a fix in one never propagated. Gate logic diverged between pr-create and checkin.

Decision

Used by 2+ skills → promote to skills/shared/; local copies forbidden; validator enforces delegation.

Lesson

Define once, maintain once. New skills reuse flags, routing, consent, severity — cheap to add.

Rejected: copy-paste ownership · big shared library up-front · "just document it".

Every run was a cold startADR 0038 · 0016

Problem

Every run started cold — the model re-read source just to orient, and scanners re-read unchanged files. Costs 80–95% too high.

Decision — orientation first

Primary: knowledge graph + architecture docs + /session-start — context arrives ready. Secondary: file-cache + scope flags (scanners only).

Lesson

Orientation beats source reads. The cache is a scoped scanner bonus — the graph is the real answer.

Rejected: re-read source every run · cold scans · "just use more context".

The manual is only 200 linesADR 0040

Problem

What's a defensible length for the always-loaded CLAUDE.md? Intuition said "fit the context window" — wrong limiter.

Decision

Target ~200 lines, justified by instruction adherence. Keep governance; externalize reference with one-line pointers.

Lesson

Capacity is ~100× headroom — a non-constraint. Signal-to-noise beats raw tokens. Same "no waste" principle as the cold-start fix.

Rejected: optimize raw tokens · re-architect the injector · "add everything while we have room".

~200 lines is what % of a 200K context window?

~1%

Capacity isn't the limiter.

Optimize for window size → wrong metric
Optimize for adherence → right metric
03

Review & trust

Correct findings, and a tool the team actually trusts.

The pass that sees spec and code togetherADR 0012

Problem

Nothing checked code as it was produced — by then the spec had dropped out of context. Code-review can only see the diff, not the intent.

Decision

The critic runs with ICEA + fresh code both in context; regenerate up to 2×, then surface. Ephemeral — no ledger.

Lesson

Only the critic can see spec and code at the same time — that's its entire reason to exist.

Rejected: skip in-generation review · let code-review catch it (too late) · fold into code-review.
ICEA
fresh code
→
critic
→
✓ passes → save
issues → regenerate ×2

No other layer can see both — it's ephemeral, no ledger.

The consent model that isn't shadyADR 0013

Problem

Skills read source without telling you — silent controller reads, PR review opening every file before you knew it started.

Decision

Three consent categories per skill: A announce · B ask-per-file (file, why, cost) · C never reads source. Every skill declares one.

Lesson

"For a tool the team is supposed to trust, opacity is poison." The cost of asking is worth the trust.

Rejected: ask-once-per-session · blanket opt-in · no consent.
A
Announce
say what you'll read and why, before reading
B
Ask per file
file name · why · token cost — your call
C
Never reads source
orientation only (graph + arch docs)

The business knows what "medium" meansADR 0015

Problem

A CVSS "medium" exposing immigration IDs or privileged matter isn't medium here. Generic scores don't know the business.

Decision

Seven triggers (B1–B7) escalate one band across all review skills; visible in the ledger; dismissible with justification.

Lesson

Domain context is non-optional; severity without it is folklore. Extensible — new categories follow B8+.

Rejected: per-skill hard-coded overrides · manual bumps · ignore context.

Evolution & the themes

How the decisions compounded — and what generalizes.

How the decisions compounded — part 1

1.27
VCS-aware ignore file — TFVC repos leaked generated files; credential protection now works on Git and TFVC.
1.31
Write gate + hierarchical ICEA — a moment to review every write; slug collisions on re-run gone.
2.0
Session-independent workflow — approve/implement/status from disk; action a Teams approval with one phrase.
2.1
Single-responsibility + draft-then-save — review plans before ICEA, ICEA before Tech Spec; laser-focused skills.

How the decisions compounded — part 2

2.6
Knowledge graph + /graph-sync — stop re-reading the whole codebase every morning.
3.0
Graph = single orientation layer — domain-map retired; graph committed & PR-reviewed.
3.3
graph.json authoritative — typed edges + confidence; module-wide fingerprints fix staleness blindness.
3.5→3.6
Deterministic EXTRACTED edges — imports parsed by script; source never enters model context.

The arc: prose orientation → structured, deterministic, committed data.

Themes that generalize — 1

01

Enforcement is a ladder

Prompt → hook → server-side. Each rule at the lowest tier that can hold it.

02

State belongs on disk

Chat is ephemeral; resumable workflows need on-disk truth.

03

One source of truth

Shared specs, one graph, one status line — fix a bug once, not three times.

04

Mechanical work → scripts

Anything a parser can resolve is cheaper and more correct outside the model.

Themes that generalize — 2

05

Signal-to-noise > token count

Capacity is abundant; adherence is the real limiter (ADR 0040).

06

Domain context is non-optional

Generic severity is folklore; the business defines "critical".

07

Transparency is table-stakes

Consent looks like friction but builds the trust adoption needs.

08

Gate your own work

The gate changes behavior by incentive, not instruction.
