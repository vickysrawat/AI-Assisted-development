In one minute: what it is

An ICEA-driven workflow for Claude Code — no code until a spec is approved — backed by a shared knowledge graph, tracked security & quality findings, and cross-session memory.

🧪 Lens 01

QA & Test Engineer

"Does the code do what was specified — and can I prove it?"

What QA worries about → what answers it

✕ The worry
  • Are acceptance criteria actually testable?
  • Does this PR really implement the spec?
  • Are we catching regressions — or re-finding them late?
  • Which findings are truly blocking?
✓ What the plugin does
  • ICEA ACs must be testable — the critic flags vague ones/critic icea
  • Traceability matrix: every change → its AC/pr-spec-review
  • Ledger marks regressions with who/when from git/code-review
  • B1–B7 severity + one pre-commit gate/checkin
→
🧪 A moment QA will recognize

The gap is visible before merge

  • /pr-spec-review maps every change to an acceptance criterion.
  • AC-4 is red — nothing implements it.
  • /checkin then blocks the commit for that gap.
  • Rework shows up in /sprint-metrics.

💻 Lens 03

Developer

"Start fast, follow a clear loop, don't re-scan what I didn't touch."

What Developers worry about → what answers it

✕ The worry
  • What am I on, and what's the context?
  • How do I go request → merge cleanly?
  • Where do things live in this codebase?
  • Why re-scan files I didn't touch?
✓ What the plugin does
  • One-pass warm-up: stack, last decision, graph/session-start
  • Guided loop; source writes gatedAPPROVE → IMPLEMENT → /checkin
  • Knowledge graph + arch docs — context arrives ready/explain · /graph-viz
  • Cache + scope flags skip unchanged files (scanners)/code-review --pr
→
💻 A moment Developers will recognize

Guardrails, not friction

  • /session-start — 3-second brief, you're oriented.
  • IMPLEMENT won't write source until you approve the diff.
  • /code-review --pr scans only the diff — seconds, not minutes.
