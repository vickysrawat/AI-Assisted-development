I
Act I · The First Absence

Structure Around the Spec

"We had code. We didn't have agreed specs."

ADRs 0001 · 0002 · 0005 · 0006 · 0010 · 0028

The failure mode: code before agreement

  • PRs bouncing for spec misalignment — spec was in Slack, code was in the PR, review was the first time anyone compared them.
  • Business logic in controllers. The AI wrote confidently in the wrong direction without anyone stopping it.
  • Same question, different answers. Closing a session meant losing context — the next session might recommend the opposite architecture.
  • Rework was the dominant cost. Not bugs, not tech debt — rework from thin or absent specs.
The ROI thesis

"One well-written spec prevents roughly three PRs that would have bounced."

ADR 0001 — the incentive math that justified the gate.

A gate is an incentive structure, not a control

The insight · ADR 0001
"Developers who know their PR will be blocked on a thin spec self-invest in spec quality before anyone else has to. The gate changes incentives — not permissions."
Output-gated, not input-triggered: the constraint fires on what the AI produces — making it structurally harder to bypass than a prompt instruction.
ADR 0001
The honesty pivot · ADR 0009 → ADR 0010
"The floor governs the willing and makes bypass visible — not impossible."
Server-side enforcement was the goal (ADR 0009). Deployment reality: the plugin never runs in CI. The answer wasn't to pretend — it was to make bypass visible via 1p checks, pr-create re-scans, and history mining. Honesty over overclaim.

II
Act II · The Second Absence

Trust Without Measurement

"We had gates. We didn't know if the AI's judgment behind those gates was right."

ADRs 0011 · 0012 · 0013 · 0015 · 0023 · 0047

The failure mode: judgment without a second opinion

  • No layer saw spec and code together. ICEA review happened before code. Code review happened after. Nothing saw both simultaneously — the critical window was empty.
  • Generalist output on specialist artifacts. A security review written through a helpful-assistant lens misses what a security engineer would flag immediately.
  • Trust was asserted, not measured. "The AI did a good job" was the entire feedback loop. No mechanism to build or calibrate trust over time.
  • Every task routed to the same model. Spec drafting, quick fixes, and code review — identical routing, regardless of what each task actually needed.

The critic: the only layer that sees both

Aha moment · The Critic
"The critic runs mid-generation — the only moment both ICEA and fresh code are simultaneously in context. Later reviews can't reconstruct that."
REVISE up to 2×, then surface to developer. Ephemeral by design — no ledger, no fingerprint. It catches spec-to-code misalignment; code-review handles code quality. Two different jobs, two different tools.

Two axes: capability and role

Aha moment · Expert Personas
"Model tier is a capability axis. Expert Personas add the role axis — orthogonal and independent. A Tech Lead and a Product Owner use the same model tier but evaluate completely different concerns."
Three constraints: stack-agnostic (same [TL] reasons across Angular, React, ASP.NET) · governance-subordinate (personas change what to scrutinise, never license assumptions) · reasoning-only (never appear in output artifacts).

Transparency, routing, and domain-aware severity

Source-file consent · ADR 0013
"For a tool the team is supposed to trust, opacity is poison."
Category A: announce file list before reading. B: ask per file — reason, what you're looking for, token cost. C: never reads source files.
ADR 0013
Multi-model routing · ADR 0023
"Route by task criticality, not by authority."
ICEA_MODEL (Opus) · REVIEW_MODEL (Sonnet) · INFRA_MODEL (Sonnet). Override per project. Both axes — capability and persona — configure independently.
ADR 0023
Business-context severity · ADR 0015
"CVSS doesn't know your domain."
B1–B7 triggers escalate findings to Critical: immigration IDs, privileged client data, vulnerable clients. Context raises severity — never lowers it.

III
Act III · The Third Absence

Continuity Between Sessions

"We had quality gates. Every session started from zero."

ADRs 0007 · 0031 · 0032 · 0034 · 0035

The failure mode: knowledge that doesn't survive

  • Closing the session reset everything. State lived in conversation context. No tab, no state. Every morning was a cold start.
  • Approval from email couldn't be actioned. The Tech Lead replied, but the plugin had already forgotten the context.
  • Memory accumulated but didn't consolidate. A subtly wrong fact promoted early steered every future session in the wrong direction.
  • No cross-session architectural memory. Decisions made Monday didn't survive to Friday's session.
The memory problem

"Memory without consolidation becomes noise."

Facts need confidence scores. Stale facts need to decay. Incorrect facts need rollback. Just like sleep consolidates what matters and discards what doesn't.

State on disk: the session-independence pivot

Aha moment · Disk-based state
"The Status line on disk is the single source of truth — not the conversation. Closing a session doesn't reset the workflow. APPROVE ADO-1847 typed in Teams, actioned immediately."
State machine on disk: DRAFT → DRAFT—Revising → ✅ Approved → IN PROGRESS → COMPLETE. REVISE resets to DRAFT and re-blocks the gate.
ADR 0031
Keyword handlers · ADR 0032
"Recognised in any message, any channel — no slash command needed."
APPROVE ADO-{ID} · IMPLEMENT ADO-{ID} · REVISE ADO-{ID} · STATUS ADO-{ID}. ADO ID normalised: ADO-1847, ADO #1847, and 1847 all work. Approval from email or Teams is actioned the moment it's typed.
ADR 0032

Dream: memory that consolidates like sleep

Aha moment · Dream Memory
"Memory accumulated but didn't consolidate. Facts decayed differently. We needed something like sleep — periodic consolidation, promotion, decay."
Runs every 5–8 sessions. Cap: 20 promoted facts in MEMORY.md. Remainder demoted to topic-*.md, loaded on demand. Append-only audit log. Rollback available. Observable quality proxies: citation stamps · [CORRECTION] events · rollback rates.

Act IV · The Fourth Absence

Structured Knowledge of the Codebase

"We had continuity. We still re-read the codebase every session."

ADRs 0016 · 0017→0038 · 0039 · 0041 · 0042 · 0043

The token efficiency problem no one warned about

Three months in, a token analysis surfaced an uncomfortable number. Every skill was treating every run as a cold start. Code review re-scanning all 50 files daily even though 48 hadn't changed in a week. The security skill loading Python, JavaScript, and Java reference material for a pure C# codebase.

From description to structure

ADR 0017 → ADR 0038 · Domain-map retired
"The knowledge graph is the single orientation layer. Typed nodes, typed edges, confidence levels — EXTRACTED (mechanically known) · INFERRED (model judgment) · AMBIGUOUS (needs human). Module-wide fingerprints for staleness."
graph.json is the structure of record. Markdown projections (index.md, per-module .md) are generated from it. All ~20 former domain-map consumers migrated to read the graph. Committed and PR-reviewed.

Mechanical knowledge should be mechanical

Aha moment · Deterministic edge extraction
"What a parser can know, a parser should know. extract-edges.js handles imports, usings, and requires — offline, no API cost, deterministic. Model judgment reserved for genuinely ambiguous relationships."
Supports: JS/TS relative imports · Python dotted + sys.path · C# using + .csproj · Java import + package. Output: merge-friendly JSON, stable across runs. Model writes only INFERRED/AMBIGUOUS edges (DI, dynamic wiring, prose-only connections).
ADR 0041
Before — model-derived EXTRACTED edges
  • Non-deterministic — same file, different edges across runs
  • Token-expensive — model call required per module
  • Approximate — may miss transitive or multi-file imports
After — extract-edges.js
  • Deterministic — identical output for identical input
  • Free — Node stdlib only, zero API tokens, offline
  • Accurate — resolves actual import graph per language

Act V · The Fifth Absence

Enforcement That Holds Under Pressure

"We had structure. We still couldn't prove it was holding."

ADRs 0004 · 0014 · 0015 · 0019 · 0020 · 0045

Persistent ledgers with FP-fingerprints

Aha moment · Finding ledgers
"Dismissal is a claim about specific code. Code changes invalidate the claim — so dismissed findings re-open automatically when the file changes."
Uniform schema: Open · Fixed (who/when/git ref) · Dismissed (justification + reason category) · Baseline (pre-existing, never gates) · Regressions. /fix FP-xxxxxxxx searches all three ledgers, applies remediation, moves finding to Fixed.
ADR 0014
code-review

code-review-ledger.md

Static analysis findings. FP-fingerprinted. Cache-aware rescan — only touches changed files.

security

security-ledger.md

OWASP/CWE/CVSS findings with B1–B7 business-context escalation applied.

dynamic-scan

dynamic-scan-ledger.md

DAST runtime findings. Same schema, same gates, same /fix remediation flow.

Three scanning passes: complementary, not competing

Phase D/P hybrid + architectural risk · ADR 0019 + ADR 0045
"Deterministic and probabilistic lenses are complementary. Phase D handles what parsers know. Phase P handles intent mismatches. A final free-form pass catches what neither can see."
7-risk cap on final pass. Evidence citation required. Scope-limited to files in scan. Labeled "LLM-inferred risk hypotheses." No repeat of Phase D findings. Pre-existing findings go to Baseline — never gate on inherited debt.
ADR 0019
Phase D — Deterministic

Parsers first

SQL injection, hardcoded secrets, lint violations. Fast, reproducible, no tokens.

Phase P — Probabilistic

Model judgment

Intent mismatches, spec-to-code gaps, context-sensitive severity — where parsers fall short.

Final pass — Architectural

Hypotheses only

Race conditions, TOCTOU, multi-file data flows. Evidence required. Up to 7 risks per scan.

The plugin applied its own philosophy to itself

Deterministic where possible · ADR 0046
"dream-init was 30–60 sequential LLM tool calls before any real work began."
Collapsed to one deterministic Node.js bootstrap script. Mechanical work (stubs, hooks, state files, gitignore) done in one pass. Rules deployed last — after the architect skill actually knows the stack. Crash-safe manifest as recovery checkpoint.
ADR 0046
Honesty over overclaim · ADR 0048
"Python was a silent dependency. It returned UNKNOWN on developer machines — while reporting success."
Eliminated. One canonical resolver (plugin-state.cjs) reads the registry — O(1), no crawl, no guessing. Hard rules: never read a relative path for version info · never build a path from a version string · never crawl ~/.claude/plugins.
ADR 0048

What 48 ADRs taught us

01

Gates are incentive structures

ADR 0001
02

Honesty over overclaim

Never claim a floor the system can't hold. Make bypass visible — not pretend it's impossible.

ADR 0010
03

One source of truth per concept

Duplication is the enemy. Shared primitives, single Status line, graph as sole orientation layer.

ADR 0003
04

Measurement enables autonomy

You don't grant trust — you measure it. Accuracy trends per category unlock graduated autonomy.

ADR 0011
05

Deterministic where possible

What a parser can know, a parser should know. Model judgment reserved for genuinely ambiguous cases.

ADR 0041
06

Friction is multiplicative

Small UX wins compound across 50+ sessions per month. Auditable exceptions beat gatekeeping friction.

ADR 0036
07

Design for deployment reality, not aspirations

Server-side enforcement was the goal (ADR 0009). The org never runs the plugin in CI. Design around reality; surface the gap honestly rather than pretending the floor holds.
