Two principles behind the structure

♻️

Share & cache everything reusable

Orientation, findings, and behavior are computed once and reused — statelessness is the enemy of cost. 80–95% reduction after baseline.

🪜

Enforce mechanically, in layers

Every rule lives at the lowest tier that can hold it: prompt → hook → server-side. "Prompts propose, hooks dispose."

The rule that keeps it honestADR 0003

Problem

Three skills, three copies of every scan bug; a fix in one never propagated. Gate logic diverged between pr-create and checkin.

Decision

Used by 2+ skills → promote to skills/shared/; local copies forbidden; the validator enforces delegation.

Lesson

Define once, maintain once. New skills reuse routing, consent, severity, gates — cheap to add.

Rejected: copy-paste · big library up-front · "just document it".

Inside graph.json

{ "meta": { "version": "1.0" }, "nodes": [{ "id": "Orders", "type": "service", "fingerprint": "a3f1…" // SHA-1 }], "edges": [{ "from": "Orders", "to": "OrdersRepo", "type": "depends", "confidence": "EXTRACTED" }], "directoryCatalog": { } }
  • Typed nodes — service · repository · ui · datastore · external-api · shared-lib.
  • Typed edges — depends · calls · publishes · reads · extends.
  • Confidence — EXTRACTED · INFERRED · AMBIGUOUS.
  • Queryable — validatable, not free-text prose.

Module-wide fingerprint

Incremental refresh

Personas & consent — governed review

🎭

personas-spec — expert lenses

Each review skill adopts a named expert persona (SAST reviewer, release manager, QA lead) so it scrutinizes the right things. Personas set what to scrutinize — never a license to assume results.

🔓

source-file-consent — A / B / C

A announce before reading · B ask per file (file, why, token cost) · C never reads source. Every skill declares one; it's the transparency contract.

Together: the right expert lens, reading only what you allowed, and telling you why. "Opacity is poison" (ADR 0013).

Here's where file-cache matters — these three scanners checksum each file and skip the unchanged ones. That delta detection is the 80–95% saving, scoped to exactly these commands (ADR 0016).

One schema, five sections

## Summary — Open:3 Fixed:9 Dismissed:2 ## Open Findings ### [FP-9a3f21c8] NULL_DEREF — High Orders.cs:88 · first-seen 2026-07-01 ## Fixed Findings # who/when from git ## Dismissed # reason + verify-flag ## Regressions # fixed → reappeared ## Baseline # first run · never gates
  • FP fingerprint = hash(file + location + type + pattern).
  • Fixed carries git author / commit / date.
  • Dismissed keeps reason + justification (required).
  • Baseline never gates — boy-scout on touched files.

Why state lives on diskADR 0031

Problem

Approve/implement assumed a live session; state lived in chat, so APPROVE after reopening did nothing.

Decision

The ICEA Status: line is the single source of workflow state; skills read it from disk every invocation.

Lesson

State on disk = resumable anywhere; approval from email/Teams actionable with one phrase.

Rejected: session-only state · dual truth (chat + file).
