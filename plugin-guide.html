Context — five versions, three lessons

Why this exists

The plugin grew from a real problem: an AI assistant that was brilliant in isolation but inconsistent under pressure.

📖
The three lessons that shaped every design decision
Architecture rationale
LessonWhat happenedWhat it became
Stateless skills are expensive Code review rescanned all 50 files daily even when only 2 changed. Security loaded Python, JS, and Java refs for a pure C# codebase. Three skills, three independent scan implementations. Shared primitives layer: domain-map.md, file-cache.json, scope flags. 80–95% token savings after first baseline run.
Orientation docs beat source reads Every skill read source files to understand structure. The ICEA skill was opening half the codebase on every invocation. domain-map.md maps feature areas to entry-point files. Skills orient in one read. Source consent spec governs when source is actually needed.
The .gitignore from day one A PAT in .claude/settings.json that is not gitignored is a real credential leak. Easy to forget under delivery pressure. Plugin handles this at setup, audits it in check 9 of dream-status, and defaults to Windows User Environment Variables.
ℹ The ICEA gate changed developer behaviour not through enforcement but through incentive alignment. A bad spec meant a blocked PR. Developers started writing better IECAs because the gate made the cost of a thin spec visible — one well-written spec prevents three bounced PRs.
Step 1

Prerequisites

Confirm these are installed before touching anything else. The plugin will partially or silently fail without them.

1
Claude Code CLI
Mandatory

The plugin runs inside Claude Code. Install it globally via npm.

bash
npm install -g @anthropic-ai/claude-code
# Verify
claude --version
⚠ Without Claude Code, none of the commands, skills, or rules will load. The install scripts will fail at the final registration step.
2
Node.js ≥ 18 and Git
Mandatory

Required by Claude Code and the install scripts.

bash
node --version   # must be ≥ 18
git --version    # any recent version
⚠ The install scripts use Node.js to update settings.json. Without it the plugin installs but never registers — it will not appear in claude plugin list.
Step 2

Install the Plugin

Run the install script once per machine — not once per project. The plugin installs to user scope and is available across all your repos.

1
Windows — PowerShell (recommended)
Mandatory

Clone or download the plugin repo, then run the installer from that folder.

powershell
# If you see "not digitally signed", run this once as Administrator first:
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

# Then run the installer:
.\install.ps1
ℹ The script asks: Clone from ADO (pulls latest) or Copy from local folder (uses files you have). Choose Clone for a clean install.
2
Mac / Linux — Git Bash (no admin required)
Mandatory
bash
bash install.sh
⚠ Do not run the installer inside a Claude Code session. Run it in a plain terminal. Running inside Claude Code can cause the plugin registration to conflict with the running instance.
Step 3

Verify Installation

Confirm the plugin registered correctly before doing anything else. A broken registration means commands silently do nothing.

✓
Check plugin is registered
Mandatory
bash
# Should list ai-assisted-development
claude plugin list --scope user

# Should show all commands and skills
claude plugin show ai-assisted-development
⚠ If ai-assisted-development does not appear in the list, the registration failed. Re-run the install script. If it still fails, check that Node.js is on your PATH and ~/.claude/settings.json is writable.
✓
Open Claude Code and check the slash picker
Mandatory

Open any project in VS Code, open the Claude Code panel, and type /ai-assisted. You should see the commands appear.

ℹ Commands only appear in VS Code after /dream-init has been run in that project (Step 4). They are visible in the CLI immediately after install.

Step 5 — Required for direct PR submission

Azure DevOps PAT

The pr-create skill can submit PRs to ADO via REST API or save a complete PR draft artifact for manual submission — a PAT is only required for direct submission. Compliance checks always run on both paths.

1
Create a PAT in Azure DevOps
Mandatory for PR creation

Navigate to https://dev.azure.com/kirklandandellis/KE → profile avatar → Personal access tokens → + New Token.

settings
Name       : claude-code-pr-create
Org        : kirklandandellis
Expiration : 30–90 days
Scope      : Code → Read & Write  (custom defined, nothing else)
⚠ ADO displays your token once only. Copy it immediately before closing the dialog.
2
Store the PAT — Option A · Windows env var Recommended
Safest option

Win+S → "environment variables" → User variables → New → AZURE_DEVOPS_PAT = your token → restart VS Code. The PAT is never written to disk in the project directory — zero accidental-commit risk.

ℹ This is the recommended default for all developers. The PAT survives VS Code restarts and can never be accidentally committed to source control.
2
Store the PAT — Option B · .claude/settings.json
Alternative — only if gitignored

Add to .claude/settings.json in your repo root. Only use this option if the file is already in .gitignore.

json
{
  "env": {
    "AZURE_DEVOPS_PAT": "your-token-here"
  }
}
⚠ Never commit .claude/settings.json. Run /dream-status after setup — it will flag a ❌ Red credential risk if settings.json is present but not gitignored. Fix that before using this option.
ℹ PATs expire. When yours expires: ADO → Personal access tokens → find the expired token → Regenerate → update your storage location.

New in v1.5.0 — Undo a Dream run

Dream Rollback

If a Dream consolidation run promoted incorrect information, wrongly deleted a memory, or made changes you want to undo, /dream-rollback lets you reverse it. The rollback is logged in memory/dream-log.md and is itself reversible.

↩
Run dream-rollback
When a Dream run needs undoing
claude code
/ai-assisted-development:dream-rollback

It will interactively guide you through:

  1. List the last 10 dream runs with change summaries
  2. Show the before/after for every ADD, UPDATE, and DELETE in the selected run
  3. Confirm before reversing — supports full or partial rollback
  4. Append a logged rollback entry to dream-log.md
ℹ Partial rollbacks are supported — you can choose to reverse only specific operations from a run. If a file was manually edited after the Dream run, dream-rollback will warn you before touching it.
Maintenance — every 5–8 sessions

Run Dream

Dream consolidates what Claude learned across your sessions into persistent memory files. Without it Claude starts each session with no knowledge of past decisions.

~
Run after every 5–8 sessions
Recommended Every 5–8 sessions
claude code
# Consolidate memory
/ai-assisted-development:dream

# View health dashboard
/ai-assisted-development:dream-health

Dream works in 6 phases: session discovery → inventory → confidence scoring → propose changes → tiered approval → audit log. Every proposed change requires your approval before anything is written.

⚠ If you never run Dream, Claude re-establishes architectural context from scratch each session through longer prompts. Session costs increase over time. Pattern knowledge (what worked, what failed, conventions confirmed) is permanently lost.

⚡
Token budget guard — Phase 0 pre-flight
Automatic

Before searching sessions, /dream checks memory size and will warn or stop if limits are exceeded:

ConditionBehaviour
MEMORY.md > 200 lines⚠ Warning — archive promoted entries to topic files first
Topic files > 10⚠ Warning — consider merging related topics
Session window > 30 conversations⚠ Capped at 30 most recent, with warning
Estimated combined context > 80K tokens❌ Hard stop — archive topic files or use a narrower keyword search
ℹ If you hit the hard stop, either archive older topic files or run /dream with a specific keyword to process one topic area at a time (e.g. say "dream — only consolidate sessions about authentication").

Security Review

Two major additions in v1.7: a static asset directory audit that runs before all other analysis, and a business context severity system that overrides CVSS scores for this domain's specific data.

0.5
Static asset directory audit — always runs first
Before SAST

Before any code analysis, the security skill enumerates public/, wwwroot/, assets/, and static/ directories and classifies every file inside them. Any .json, .csv, .xlsx, or data file containing real entity data is flagged as CRITICAL immediately — before other findings are shown.

⚠ Why this matters for this project: Angular's public/ directory is copied verbatim into the build output and served at the root URL with no authentication. Any file here is accessible at http://[server]/filename with no login required. Real client names, A-Numbers, or matter data committed here constitutes an immediate data breach.
ℹ This check also runs inside /checkin — staged files in static directories containing real entity data always block the commit.
B1–B7
Business context severity override
All review skills

CVSS is a mathematical floor — business context can raise severity, never lower it. All review skills (security, code-review, icea-review, checkin, pr-spec-review) apply the override triggers from skills/shared/business-context-severity.md.

TriggerData typeWhy Critical
B1Attorney-client privileged dataBar ethics rules — privilege cannot be restored once breached
B2Immigration identifiers (A-Numbers, visa numbers, case IDs)Active proceedings at risk, mandatory breach reporting
B3Hearing dates, filing deadlines, case statusMissed deadline = irreversible harm in immigration proceedings
B4Pro bono / vulnerable client dataIndividuals with no recourse, life-altering consequences
B5Data triggering mandatory breach notificationState privacy laws, bar ethics reporting obligations
B6Physical safety dataIdentity/location for individuals at risk of harm
B7Real PII in a static-serving directoryDirectly accessible by URL, no authentication required

Design Philosophy

One principle governs everything the plugin generates, reviews, and validates: do not over-complicate. Prefer the simplest solution that correctly solves the problem. This is enforced globally via rules/project-rules.md (active on all files) and is always present in CLAUDE.md.

🎯
Five principles — enforced as project rules
Applies to all files
PrincipleWhat it means in practice
Simplicity If a simpler approach exists, take it — even if it means more lines. Complexity without a concrete requirement is a defect, not a feature.
Readability Code is read far more often than written. Optimise for the reader, not the writer. Self-documenting names over clever shortcuts.
Maintainability Explicit and self-contained over clever abstractions. A future developer should understand and change any piece without reading the whole codebase.
Testability No hidden side effects, no global state, no deep coupling. Code that cannot be unit-tested without heroic effort is a design smell.
Do not assume If a requirement, instruction, or expected behaviour is vague, ambiguous, or complex enough to support more than one interpretation — stop and ask. Never guess and proceed. A wrong assumption costs more to unwind than the question costs to ask.
Decision transparency For any complex or non-trivial logic, document the decision inline: list the options considered, why each was rejected, and why the chosen approach was selected. A future developer should never have to guess why something was built the way it was.
⚠ The code review and ICEA skills will flag violations — over-engineered abstractions, unnecessary generics, premature optimisation, and deep coupling are treated as findings, not style preferences. The canonical statement in project-rules.md: "Complexity that cannot be justified by a concrete requirement is a defect."
📝
Decision transparency — required for complex logic

Whenever a non-trivial design choice is made, document it inline above the implementation. List the options considered, why each was rejected, and why the chosen approach was selected.

typescript · inline comment format
// Design decision: character-count cache for file change detection
// Options considered:
//   A) Git diff (git diff --name-only)
//      Rejected: requires git CLI, fails on untracked new files
//   B) File hash (MD5/SHA256)
//      Rejected: slower than char count, overkill for change detection
//   C) Character count  ← chosen
//      Selected: fast, no dependencies, sufficient for detecting edits
ℹ Apply this to any logic a future developer would reasonably ask "why was this done this way?" — algorithm selection, data structure choice, pattern selection, or architectural approach.
ℹ Defined in two places for maximum coverage: rules/project-rules.md (activates on **/* — every file) and CLAUDE.md Section 3 (always in session context regardless of active skill).

Source file consent — second governing principle
New in v1.8.1

Every skill that reads source files operates under one of three consent categories, defined in skills/shared/source-file-consent.md. Claude never reads a source file without telling you which file, why it needs it, what it is looking for, and how much it will cost.

CategorySkillsBehaviour
A — Implicit /code-review /security-review /checkin Developer consented by invoking the command. Skill announces scope before first file read. No confirmation prompt — type STOP to halt.
B — Explicit gate icea-review pr-spec-review /bug /update-arch /explain Before reading any file beyond the diff or architecture docs, the skill asks: which file, why, what it is looking for, token cost. You say yes / no / read a different file.
C — Never session-start dream-status dream icea-feature Hard rule — these skills never read source files regardless of what the developer asks. Architecture docs and memory only.

Model Routing

The plugin routes tasks to different models based on their nature. Generative work (planning, coding) uses a more capable model. Analytical review work uses a faster model — reducing cost and timeout risk.

⚙
Two routing tiers
Configurable per project
TierDefault modelSkillsWhy
Generation claude-opus-4-6 icea-feature, ado-tasks, pr-describe, product-docs Quality at generation time reduces rework downstream
Review claude-sonnet-4-6 icea-review, code-review, security, pr-spec-review, pr-create gate, critic 5× faster than Opus — reduces connection timeout risk on long scans

Override either tier in .claude/settings.json without editing any skill files:

json · .claude/settings.json
{
  "env": {
    "AZURE_DEVOPS_PAT": "your-pat-here",
    "ICEA_MODEL":       "claude-opus-4-6",
    "REVIEW_MODEL":     "claude-sonnet-4-6",
    "INFRA_MODEL":      "claude-sonnet-4-6",
    "CRITIC_MODEL":     "claude-sonnet-4-6"
  }
}
ℹ If an env var is not set, the skill uses its hardcoded default — no action needed unless you want to change the routing. See skills/shared/model-routing-spec.md for full documentation.

Extending the plugin · for contributors

Adding a Skill

Skills trigger automatically on keyword match. The dynamic-scan skill (added in v1.10) is a recent worked example — a SKILL.md orchestrator plus a references/ folder of lazily-loaded detail files.

1 — Create the folder

skill folder
skills/<skill-name>/
├── SKILL.md
└── references/          ← optional; only if the skill needs lazily-loaded data

2 — Write the frontmatter

The YAML description is what Claude matches against at invocation time — precision reduces false triggers.

yaml · SKILL.md frontmatter
---
description: >
  One or two sentences on what this skill does and when to use it.
  End with trigger phrases: "phrase one", "phrase two", "phrase three".
---

3 — Write the body (conventions)

  • Open with a purpose statement — what it does and what it does not do.
  • Put hard rules in a closing section — constraints the skill must never break regardless of user input.
  • Add a model-routing note citing ../shared/model-routing-spec.md and state the tier (generation / review / infrastructure).
  • If it reads or writes a shared cache, cite ../shared/single-writer-assumption.md.
  • Reference ../shared/business-context-severity.md — the structural validator requires every SKILL.md to reference it.
  • Large or conditional data goes in references/, loaded lazily in the relevant step — never all at skill start.

4 — Register and wire up

  • Add the name to components.skills in plugin.json.
  • If slash-invokable: create skills/command-stubs/<name>.md and a commands/<name>.md, then add the stub to the dream-init deploy loop and the dream-status stub check.
  • Add the skill to the Category A/B/C table in shared/source-file-consent.md.
  • Create tests/skill-scenarios/<name>.yaml — at least one trigger-detection scenario and one hard-rule scenario.
ℹ Run the validator. python3 tests/validate.py --strict must pass with zero errors before the skill ships. It cross-checks the manifest, the stub loops, the consent table, the CHANGELOG version, and more.

Adding a Command

Commands live in commands/ and are always invoked explicitly — no keyword trigger detection. Most commands are a thin wrapper that resolves flags and then executes a skill in full.

markdown · commands/<name>.md
---
description: One-line summary shown in the VS Code command picker.
argument-hint: [--flag-one | --flag-two]
---

# /command-name

What this command does and when to run it. Resolve flags, then read
skills/<skill>/SKILL.md and execute it from Step 0 to the end.

Then: add the name (without /) to components.commands in plugin.json, create a matching stub in skills/command-stubs/, and update dream-init (deploy loop + table + Step 10 summary) and the dream-status stub count so the new stub is tracked.

Extending the plugin · for contributors

Releasing a Version

  1. Make all changes on a feature branch.
  2. Prepend a ## [X.Y.Z] — YYYY-MM-DD section to CHANGELOG.md — list affected files and the exact behaviour change.
  3. Bump version in .claude-plugin/plugin.json.
  4. Update the version badge in README.md and the Plugin version: line in CLAUDE.md.
  5. If model defaults changed, update recommended_models and the model-routing references.
  6. Run python3 tests/validate.py --strict (structural) and node tests/runner.js (behaviour); confirm both pass.
  7. Open a PR to main titled [vX.Y.Z] Short description. After merge, tag: git tag vX.Y.Z && git push --tags.
Change typeVersion bump
New skill, new command, new shared specMinor (X.Y → X.Y+1)
Bug fix, spec update, new checker, documentationPatch (X.Y.Z → X.Y.Z+1)
Breaking change to a shared spec or cache schemaMajor (X → X+1)

The v2.6.0 release adds the codebase knowledge graph (.claude/graph/), the /graph-sync command and skill, git-hook stale-detection, two new shared schemas (graph-index-schema.md, graph-module-schema.md), write-gate diff-only display, and output-token reduction via customInstructions. It is a representative minor bump: new skill + command, manifest updated, stubs extended (36 total), CHANGELOG entry added, validators green. The preceding v1.26.0 release added the dream-audit skill and enforcement-floor hooks in the same shape.
