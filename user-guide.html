Plugin User Guide

ai-assisted-development

ICEA-driven development workflow for distributed teams using Azure DevOps — language-agnostic across .NET · Java/Spring Boot · Python · Node.js backends and Angular / React frontends. Every feature starts with a spec. Every PR is compliance-checked before it opens.

v1.26.0 .NET 8 Angular 17+ Node.js Java / Spring Python Azure DevOps 12 health checks

What is this plugin?

This Claude Code plugin enforces a formal development workflow on top of your existing team processes. It intercepts feature requests, forces a spec (ICEA) before any code is written, runs cache-aware code and security reviews, and creates Azure DevOps pull requests with compliance gates built in.

The core principle behind every skill and rule:

◈Do not over-complicate. Prefer the simplest solution that correctly solves the problem. Complexity without a concrete requirement is a defect.

The ICEA workflow in one sentence

Before any code is written, Claude creates an Intent · Context · Examples · Acceptance document, waits for your explicit approval, then generates implementation in the correct layer order with a PR description pre-filled and ready to submit.

What the plugin does not do

  • It does not modify your source files without your knowledge
  • It does not create or merge PRs without explicit confirmation
  • It does not run any external scans — all analysis happens in-context
  • It does not write to MEMORY.md or topic files without tiered approval

PowerShell — Windows (recommended)

.\install.ps1

If you see "not digitally signed":

Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

Both scripts prompt you to clone from Azure DevOps or copy from a local folder. They handle everything: registering the marketplace, updating settings.json, installing to user scope.

Building features (ICEA gate)

The icea-feature skill activates automatically when you describe something to build. It intercepts the request and creates a full ICEA document before writing any code.

✓Why ICEA first? A thin spec causes rework. An approved ICEA becomes the contract that code review, PR review, and sprint metrics all check against.

How the gate fires

The ICEA gate is enforced in two layers:

  1. Session-start pre-load — /session-start loads the feature gate rule into active context at the beginning of every session, so Claude knows the constraint before any message is processed.
  2. Output constraint in CLAUDE.md — a standing rule prevents implementation code from being written for new behaviour without an approved ICEA on disk at docs/icea/ADO-[ID]-*.md.

Team convention: invoke /ai-assisted-development:icea-feature explicitly before describing a feature. This is the reliable path — do not rely solely on the gate firing reactively.

⚠Always run /session-start at the start of each session to ensure the feature gate rule is active. Sessions started without it fall back to the CLAUDE.md output constraint only.

The flow

⏸ ICEA gate → DRAFT → approval → implementation
1
Gate fires

Claude responds with ⏸ ICEA GATE — Feature detected. before doing anything else.

2
Draft created → critic (gate 1)

Claude generates the full ICEA from the knowledge graph and your description. The critic then reviews the draft for completeness, testability, B1–B7 coverage, and scope — its concerns are folded into the [?] gap list so you see everything in one place.

3
You review and approve

Reply APPROVED to proceed. Use EDIT [section] to change a specific part. Use REJECT [reason] to start over.

4
Implementation generated → critic (gate 2)

In dependency order for the layers your repo actually has — persistence → service → API → UI → tests → PR description — in the project's detected stack (.NET, Java/Spring Boot, Python, or Node.js). The critic reviews the generated code before any file is written to disk. A clean verdict writes the files; a REVISE verdict regenerates (up to twice) before surfacing to you.

⚠The word APPROVED is required exactly. "Looks good", "fine", or "go ahead" will not trigger code generation — the approval block will remain visible.

Decisions and the change manifest (v1.26.0)

For complex work, the ICEA now carries two additions. A D — Decisions block appears when the implementation faces a genuine fork (new dependency, schema choice, placement question): each option carries a "choose this when" case, the recommendation cites this codebase's evidence, and approval is a selection — OPTION A, OPTION B, EDIT, or DIRECT with your own approach. Your tech lead approves the approach they can actually see. A Change Manifest lists every file the model predicts it will touch and why, before any code exists. At checkin the prediction is scored against the real diff; deviations feed the memory system so future predictions improve. The manifest measures predictability — it never blocks, and it is not a correctness claim.

Change tiers (v1.26.0)

Not every change gets the full ceremony. The plugin classifies each change mechanically — you never pick a tier:

  • T1 micro (≤1 file, ≤20 lines, no signature changes, no new dependencies, no sensitive paths): a one-line auto-ICEA is written and work proceeds immediately. The critic still reviews the output. Say STOP to force a full ICEA.
  • T2 standard: the full ICEA draft and APPROVED cycle you know.
  • T3 elevated (auth paths, public endpoints, new dependencies, sensitive data, schema changes): full ICEA plus mandatory permission-boundary examples.

If a T1 change grows beyond its bounds, /checkin blocks the commit and routes you to a full ICEA — tiers only move up, never down.

Mechanical enforcement (v1.26.0)

As of v1.26.0 hooks install by default during dream-init (opt-out via --no-hooks is recorded and attributed), and the authoritative floor moves server-side: validate-pr-compliance.py runs as a required branch-policy check, verifying an approved ICEA exists per PR and re-checking T1 bounds as pure diff math — unbypassable from any client. Hook integrity is monitored (dream-status check 1p) and re-synced on every version bump. Four deterministic hooks back the prompt-level gates: a write-blocker when no approved ICEA exists, a git pre-commit hook that blocks commits with open Critical/High findings even if you bypass /checkin, and a CI script that fails the pipeline on ledger integrity violations (e.g. a dismissal with an empty justification). dream-init offers all three during setup; see hooks/README.md.

Emergency bypass

Prefix your message with /skip-icea to bypass the gate. Claude will warn once and then proceed. This is logged and not recommended for regular use.

The critic layer

The critic is a second-pass reviewer that checks an artefact at the moment it is produced and before it is saved. It runs automatically inside the ICEA flow at two points, and you can also run it on demand with /critic.

✓Why a critic, when there is already code review? Code review runs on files that already exist — by then the spec is out of context. The critic runs while the ICEA and the freshly generated code are both in front of Claude, so it can ask the one question a later scan cannot: does this do exactly what the approved AC asked — no more, no less?

The two automatic gates

GateWhenWhat it checksWhat a problem does
Gate 1 — ICEAAfter the ICEA draft, before you approveCompleteness, testability, B1–B7 coverage, scope vs IntentConcerns fold into the gap list you review before approving — no separate step
Gate 2 — codeAfter code is generated, before any file is writtenICEA traceability, simplicity, rules compliance, decision transparency, hidden assumptionsA REVISE verdict regenerates the code (up to 2 automatic attempts) before anything reaches disk

What the code gate does on a REVISE verdict

🔎 generate → critique → (revise ×2) → write or surface
1
Critique

The critic reports a verdict: PASS, PASS WITH NOTES, or REVISE. PASS verdicts write the files immediately.

2
Auto-revise (up to 2×)

On REVISE, Claude announces 🔁 Critic revision 1 of 2, regenerates the affected code addressing the concerns, and re-critiques. If a retry produces the same concerns with no movement, it surfaces early rather than spending the second attempt.

3
Surface to you

If it still fails after two retries, you choose: ✅ ACCEPT AS-IS (write despite the notes), ✏ GUIDE (tell it what to change), or ❌ HALT (write nothing).

⚠While the verdict is REVISE, nothing is written to disk. The ICEA file itself is still saved on approval — only the implementation code is gated by the critic.

Running it yourself

Use the command directly when you already have an approved ICEA or already-written code and want a focused second opinion:

bash/critic icea ADO-1847    # critique an existing ICEA spec
/critic code             # critique your staged / changed code
/critic                  # infer the phase from context
ℹThe standalone /critic code path reports concerns only — it never rewrites files, because the code is already on disk. To apply a remediation, use /fix FP-xxxxxxxx or edit manually. The automatic regenerate loop only applies to the in-flow gate, where the code is still unsaved.

What the critic does not do

The critic is ephemeral: it keeps no ledger and assigns no fingerprints. Persistent finding tracking stays with /code-review, /security-review, and /fix. The critic is a gate, not a tracker — it decides whether an artefact is good enough to save, then gets out of the way.

Model

The critic uses CRITIC_MODEL, which falls back to REVIEW_MODEL (Sonnet) when unset. To run a higher-quality critic over a Sonnet review tier, set CRITIC_MODEL to Opus in .claude/settings.json.

Code review

Coverity-style static analysis with persistent cross-run tracking. Skips unchanged files by default — token cost drops 80–95% on subsequent runs.

(no flag)Cache-aware full-project scan — skips files unchanged since last review
--changedOnly git-staged and unstaged modified files
--prOnly files changed in this branch vs the base branch
--fullForce a complete scan — ignore cache entirely
--ciCI mode: same as --full, plus warns if a cache file is found on disk
/ai-assisted-development:code-review              # default scan
/ai-assisted-development:code-review --changed    # staged/modified only
/ai-assisted-development:code-review --pr         # this branch vs dev
/ai-assisted-development:code-review --full       # force full rescan

Reports are written to CodeReviews/. A running ledger (code-review-ledger.md) tracks findings across runs, flagging new issues, resolved issues, and regressions.

Hybrid analysis (v1.26.0)

Code review now runs in two phases. Phase D (deterministic) captures findings from tools already on your machine — your project's own analyzer build warnings for C#, project-local ESLint for JS, and build-free web.config security checks. The first run records all pre-existing findings as a baseline that never gates; only new findings, or baseline findings in files you touch, can block. Phase P (probabilistic) handles judgment: spec traceability, business-context severity, cross-file reasoning. Every ledger entry shows its Source, and the scope report states exactly which tools ran on your machine — "unscanned" is never disguised as "clean".

Dynamic scan (DAST)

The runtime counterpart to security review. While /security-review reads your source code (static analysis), /dynamic-scan tests a running application or API with OWASP ZAP — it crawls the live site and, when you ask for it, sends real attack payloads to find vulnerabilities that only appear at runtime. ZAP runs in Docker, so nothing is installed permanently on your machine.

Safe by default. With no flags, the scan is passive — it observes traffic and audits your dependencies, but sends no attack payloads. Active scanning (which does send payloads) only happens when you pass --full or --scope, and the command first confirms the target is not production and that you are authorised to test it.

/ai-assisted-development:dynamic-scan --url http://host.docker.internal:5000   # passive baseline + dependency audit
/ai-assisted-development:dynamic-scan --url http://host.docker.internal:5000 --stack mvc --auth form
/ai-assisted-development:dynamic-scan --url http://host.docker.internal:5001 --stack api --swagger ./swagger.json --auth token
/ai-assisted-development:dynamic-scan --deps-only                              # dependency audit only — always safe
/ai-assisted-development:dynamic-scan --url $STAGING --full                    # active scan (gated: staging only)
/ai-assisted-development:dynamic-scan --url $STAGING --ci --fail-on High       # pipeline mode

Reports are written to dynamic-scan/dynamic-scan-<date>.html. Open in any browser.

PR workflow

Three skills work together to take a branch from ready-to-review to open PR in ADO — or to a saved draft if ADO connectivity is unavailable:

pr-describe → pr-create (compliance always) → submit to ADO or save draft
1
Generate a PR description

Say "write PR description" or "ready to merge". pr-describe generates an ICEA-compliant description with the compliance checklist pre-filled.

2
Compliance checks — always run

pr-create runs icea-review and the findings gate against all three ledgers. These checks cannot be skipped regardless of whether you submit to ADO or save a draft. Result: ✅ Ready · ⚠ Needs work · ❌ Blocked.

3
Choose: connect to ADO or save draft?

After compliance, pr-create asks which path you want. A ❌ Blocked result skips this question entirely — fix the compliance issue first.

4
Submit or save

Connect: resolves your PAT, enriches the PR title from ADO, shows a confirmation prompt, and submits via REST API. Draft: asks for Release ID and Sprint ID, writes a complete artifact to docs/Release<ID>/Sprint<ID>/pr-draft-ADO-<ID>.md with a pre-filled browser submit URL.

CI / offline mode

For pipelines or environments without interactive prompts, use the --offline flag. Both Release ID and Sprint ID are required as arguments — the skill hard-stops if either is missing:

/ai-assisted-development:pr-create --offline --release 3 --sprint 42
⚠To bypass the ICEA compliance gate in an emergency, add --skip-icea-check. To bypass the findings gate, add --skip-security-gate with a written justification. Both overrides are recorded in the draft artifact and PR description.

PR spec review

Say "review PR against spec" or "traceability matrix". pr-spec-review checks the PR diff against the ICEA and produces a structured pass/fail traceability report with gaps and risks.

Dream memory system

The Dream system maintains per-project AI memory across Claude Code sessions. Claude reads your sessions directly — you don't need to write to MEMORY.md manually.

Consolidate memory

/ai-assisted-development:dream

Run every 5–8 sessions. Claude searches conversations since the last run, scores knowledge candidates, and proposes ADD / UPDATE / DELETE actions. Nothing is written until you approve.

Token budget limits

Before searching, /dream checks the current memory size and will warn or stop if limits are exceeded:

ConditionBehaviour
MEMORY.md > 200 lines⚠ Warning — archive promoted entries to topic files first
Topic files > 10⚠ Warning — consider merging related topics
Session window > 30 conversations⚠ Capped at 30 most recent
Estimated context > 80K tokens❌ Hard stop — archive topic files or use a narrower keyword

Health dashboard

/ai-assisted-development:dream-health

Generates memory/health.html. Open in any browser to see confidence scores, decay curves, promote candidates, and clickable justification panels.

Rollback

/ai-assisted-development:dream-rollback

Reverts the last consolidation using the dream-log.md audit trail. After a rollback, dream-status will show ⚠ Amber for the rollback check until /dream is run again to re-consolidate.

Codebase knowledge graph New v2.6.0

The plugin maintains a lightweight map of your codebase under .claude/graph/: a graph-index.md listing every module, plus one small detail file per module (each capped at ~400 tokens). Skills read this map for fast orientation instead of scanning source, which keeps token cost down on large repositories.

Keeping it fresh

Each module records a fingerprint of its entry-point files. A git post-merge hook sets a stale flag when those files change, and dream-status surfaces it. Run the refresh to bring only the changed modules current — untouched modules are skipped:

/ai-assisted-development:graph-sync

graph-sync regenerates only stale detail files, detects newly-added modules, and — once a project crosses 30 modules — restructures the flat layout into a domain-grouped one. It clears the stale flag on success. After a plugin upgrade, dream-sync reminds you to run it.

Model routing

Skills route to different models based on the nature of the task. You can override defaults in .claude/settings.json.

GenerationICEA_MODEL · default: claude-opus-4-6
Used by: icea-feature · ado-tasks · pr-describe · product-docs
ReviewREVIEW_MODEL · default: claude-sonnet-4-6
Used by: icea-review · code-review · security · dynamic-scan · pr-spec-review · pr-create gate · critic (via CRITIC_MODEL, falls back to REVIEW_MODEL)
InfrastructureSonnet · no override variable
Used by: dream · architect · dream-status · sprint-metrics · token-analysis

Override in .claude/settings.json

json{
  "env": {
    "AZURE_DEVOPS_PAT": "your-pat-here",
    "ICEA_MODEL":       "claude-opus-4-6",
    "REVIEW_MODEL":     "claude-sonnet-4-6"
  }
}
⚠Never commit .claude/settings.json. It contains your PAT. Confirm it is in .gitignore before adding anything to it.
