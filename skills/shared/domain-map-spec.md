Shared by: `architect` (writer), `icea-feature`, `icea-review` (readers)

Defines the schema, authoring rules, and staleness contract for `domain-map.md`.

```markdown
_Generated: YYYY-MM-DD_
_Fingerprint: <sha1 of all entry-point files, sorted — written by architect skill>_

- Frontend: <stack description>

- **Key files**:
  - `<relative/path/FileA>` — <one-line description (≤10 words)>
  - `<relative/path/FileB>` — <one-line description (≤10 words)>
- **Notes**: <cross-cutting concerns, patterns, gotchas — omit section if none>

```

Repeat the `### <AreaName>` block for every meaningful feature area.

---

## Authoring rules (for architect skill)

- Maximum **3 files** in "Key files" per area — entry point is separate
- Omit test files, migrations, auto-generated code, and scaffold boilerplate
- If a folder has only 1–2 files, merge it with the closest parent area
- Do NOT invent feature areas — only map directories that exist
- Do NOT read file contents — derive area names from directory and file names only
- Use `PascalCase` for area names, matching the project's naming convention
- The `_Generated: YYYY-MM-DD_` line is the freshness timestamp — always write today's date

---

## Reading rules (for icea-feature, icea-review)

1. Read the `## Stack` section first — confirms which layers are present
2. Match the feature request or PR to the closest `### <AreaName>` section
3. Use the **Entry point** path as the primary reference — do not open other files
   unless the domain-map explicitly lists them in **Key files**
4. If no area matches, note it and proceed without orientation (do not scan)
5. Check the `_Generated:` date — if stale (see staleness rules below), warn and continue

## Staleness rules

A domain-map is considered potentially stale when **either** of the following is true:

**1. Structural staleness** — `domain-map.md` modification date is more than 7 days
older than the last git commit that added, renamed, or deleted a source file:
```bash
git log -1 --diff-filter=ARD --format="%ct" -- . 2>/dev/null || echo "0"
```

**2. Content staleness** — the SHA-1 fingerprint of the listed entry-point files has
changed since the map was generated. The architect skill embeds a `_Fingerprint:` line
in the map (see Schema below); readers compare it against a fresh hash of the same files:
```bash
# Extract entry-point paths from domain-map.md
ENTRIES=$(grep "^\- \*\*Entry point\*\*:" .claude/architecture/domain-map.md | sed 's/.*`\(.*\)`.*/\1/')
# Hash all entry-point files together (order-stable sort)
CURRENT=$(echo "$ENTRIES" | sort | xargs -I{} sha1sum "{}" 2>/dev/null | sha1sum | cut -d' ' -f1)
# Compare against stored fingerprint
STORED=$(grep "_Fingerprint:" .claude/architecture/domain-map.md | sed 's/.*_Fingerprint: \([a-f0-9]*\).*/\1/')
[ "$CURRENT" = "$STORED" ] && echo "FRESH" || echo "STALE"
```

If the `_Fingerprint:` line is absent (maps generated before spec v1.1), treat the map
as structurally stale and regenerate on the next architect run.

Staleness is **non-blocking** — skills warn and continue, never halt.

Warning format (structural):
```
⚠ domain-map.md may be stale (generated: {date}, last structural change: {date}).
  Run the architect skill to refresh it for accurate orientation.
  Continuing with current map…
```

Warning format (content):
```
⚠ domain-map.md entry-point files have changed since the map was generated.
  Orientation for {AreaName} may be inaccurate.
  Run the architect skill to refresh it.
  Continuing with current map…
```

## Single-writer contract

Only the `architect` skill writes `domain-map.md`. Skills that read it must
never modify it. If a reading skill discovers the map is incorrect, it should
warn the developer and suggest re-running the architect skill.
