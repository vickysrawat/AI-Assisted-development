# Graph JSON Schema
_Schema version: 1.0 · Single source of truth for `.claude/graph/graph.json`_

`graph.json` is the **machine-readable structure** of the codebase knowledge graph:
typed nodes, typed edges, and per-module fingerprints. It is the *authoritative*
representation of graph structure — the markdown files (`graph-index.md` and the
per-module detail files) are a **generated projection** of it for the model to read.

This split closes the FAIR *Interoperable* gap (relationships become data a tool can
query, validate, and visualise) while keeping the always-loaded context lean: the
markdown stays small and `graph.json` is **never auto-loaded** — it carries no
`paths:` frontmatter, so it enters context only when a skill explicitly reads a slice
of it.

---

## Location

```
.claude/graph/graph.json
```

**Committed and PR-reviewed — not gitignored.** As the authoritative structure it
belongs in version control, reviewed like source and shared across the team
(consistent with the markdown graph, [ADR 0038](../../docs/adr/0038-knowledge-graph-orientation.md)).
Generated by the `architect` skill (Step 7), refreshed incrementally by `/graph-sync`.

**Determinism (minimise merge friction).** Writers MUST emit stable output: `nodes`
sorted by `id`, `edges` sorted by `(from, to, type)`, object keys in the order below,
2-space indent, and a trailing newline. Two syncs of an unchanged repo must produce a
byte-identical file so diffs stay minimal.

## Top-level shape

```json
{
  "meta": { ... },
  "nodes": [ ... ],
  "edges": [ ... ]
}
```

### `meta`

```json
{
  "schemaVersion": "1.0",
  "generatedAt": "2026-07-03",
  "generator": "graph-sync",
  "structure": "flat",
  "moduleCount": 5
}
```

| Key | Rule |
|---|---|
| `schemaVersion` | This schema's version — `"1.0"` |
| `generatedAt` | Date of last write (`YYYY-MM-DD`) |
| `generator` | `"architect"` or `"graph-sync"` — what last wrote the file |
| `structure` | `"flat"` or `"domain"` — must match `graph-index.md` |
| `moduleCount` | Must equal `nodes.length` and the index row count |

### `nodes[]` — one per module

```json
{
  "id": "orders",
  "module": "Orders",
  "domain": "orders",
  "type": "service",
  "detailFile": "graph/orders.md",
  "entryPoint": "src/Orders/OrderService.cs",
  "paths": ["src/Orders/**"],
  "fingerprint": "a3f8c21d9e4b7062f5d1",
  "hub": false
}
```

| Key | Rule |
|---|---|
| `id` | Stable identifier — the detail-file basename without extension, lowercase-kebab. Edges reference this. |
| `module` | PascalCase display name (matches the index `Module` column) |
| `domain` | snake_case domain label (matches the index `Domain` column) |
| `type` | **Node type (required)** ∈ `service` · `repository` · `ui` · `datastore` · `external-api` · `shared-lib` · `domain`. Drives filterable traversal, precise impact analysis, and visualisation grouping. When ambiguous, pick the closest and let `/graph-sync` refine it. |
| `detailFile` | Relative path from `.claude/` — matches the index `Detail File` column |
| `entryPoint` | Most-representative file (human orientation pointer; matches the index `Entry Point`) |
| `paths` | **Array** of source-root globs the module owns. Usually one; a multi-root module (e.g. frontend + backend) lists each. Fingerprint and traversal-load span all of them. |
| `fingerprint` | **Module-wide** fingerprint — see below. Mirrored into the detail file's `_Fingerprint:` header. |
| `hub` | Derived flag: `true` when the node's total degree (in + out edges) exceeds the hub threshold. Traversal-load excludes hubs from neighborhood expansion so a `Core`/`Common` module never blows the token budget. |

### `edges[]` — typed, directed dependencies

```json
{ "from": "orders", "to": "payments", "type": "depends", "confidence": "EXTRACTED", "reason": "settlement" }
```

| Key | Rule |
|---|---|
| `from` | Source node `id` (must exist in `nodes`) |
| `to` | Target node `id` (must exist in `nodes` — **no dangling edges**) |
| `type` | Relationship type ∈ `depends` (default) · `calls` · `publishes` · `reads` · `extends` |
| `confidence` | Provenance ∈ **`EXTRACTED`** (found directly in source — e.g. an import/using/reference), **`INFERRED`** (reasonable model inference), **`AMBIGUOUS`** (flagged for human review). Impact analysis must never treat an `INFERRED`/`AMBIGUOUS` edge as fact. |
| `reason` | Optional short phrase (≤4 words) — why the dependency exists |

**Reverse edges are derived, not stored.** "What depends on X?" = every edge whose
`to == X`. Inverting the edge list is a single pass, so `dependents` is computed at
query time — never materialised (keeps the file minimal and drift-free).

## Module-wide fingerprint (correctness)

The fingerprint is a hash over **all source files under the node's `paths`**, not a
single entry-point file — so a change to *any* file in the module marks it stale
(the single-file anchor silently missed those). Compute it identically everywhere it
is produced (`architect`, `/graph-sync`, and `hooks/graph-stale-detect.sh`):

```bash
# graph_module_fingerprint <path-root> [<path-root> ...] -> sha1
graph_module_fingerprint() {
  { for root in "$@"; do
      find "$root" -type f \
        -not -path '*/.git/*'   -not -path '*/node_modules/*' \
        -not -path '*/bin/*'    -not -path '*/obj/*' \
        -not -path '*/dist/*'   -not -path '*/.angular/*' \
        -not -path '*/migrations/*' -not -path '*/__pycache__/*' \
        -print0 2>/dev/null
    done; } \
  | sort -z \
  | xargs -0 sha1sum 2>/dev/null \
  | sha1sum | cut -d' ' -f1
}
```

- Hashes both filenames and contents (each `sha1sum` line is `hash␠␠path`), so adds,
  deletes, renames, and edits all change the fingerprint.
- `sort -z` makes it order-independent and deterministic.
- The ignore globs are the **intra-module noise floor** (generated code, migrations,
  build output, vendored dirs) so churn in those files does not force regeneration.
  `/graph-sync` may extend them from the detected stack.

## Integrity rules (enforced by `tests/validate.py` check 9)

When `graph.json` exists it must satisfy:

1. `meta.moduleCount == nodes.length ==` the number of `graph-index.md` rows `==` the
   number of detail files (**no orphans**: no index row / node / detail file without
   the other two).
2. Every `edge.from` / `edge.to` resolves to a node `id` (**no dangling edges**).
3. `meta.structure` matches the index header's `Structure`.
4. Node `id`s are unique.
5. **Projection agreement:** each node's `fingerprint`, `detailFile`, `entryPoint`,
   `module`, and `domain` match the corresponding detail file and index row (**no
   drift** between the JSON source and its markdown projection).

**Cycles are reported, not rejected.** Legitimate bidirectional dependencies exist;
`/graph-sync` and validation *warn* with the cycle path and honour an allowlist rather
than hard-failing.
