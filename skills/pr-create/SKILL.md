## Purpose

Submit a Pull Request to Azure DevOps using the ADO REST API, or produce a
complete PR draft artifact for manual submission. This skill completes the
end-to-end PR workflow: `pr-describe` → `pr-create` → `pr-spec-review`.

Compliance checks and the findings gate **always run on both paths** — no PR
artifact is produced without passing (or explicitly acknowledging) these gates.

**Requires:** An ICEA-validated PR description already generated by the `pr-describe`
skill in this conversation. Do not create PRs without a validated description.

---

## Flag handling — check before Step 1

```bash
# Parse flags from the invocation arguments
OFFLINE=false
RELEASE_ID=""
SPRINT_ID=""
SKIP_ICEA=false
SKIP_SECURITY=false

for arg in "$@"; do
  case "$arg" in
    --offline)          OFFLINE=true ;;
    --release)          RELEASE_ID="$2"; shift ;;
    --sprint)           SPRINT_ID="$2"; shift ;;
    --skip-icea-check)  SKIP_ICEA=true ;;
    --skip-security-gate) SKIP_SECURITY=true ;;
  esac
  shift
done
```

**If `--offline` is present but `--release` or `--sprint` is missing — hard stop:**

```
❌ --offline requires --release <ID> and --sprint <ID>
   Example: /pr-create --offline --release 3 --sprint 42
```

Do not continue. Exit immediately.

## Step 1 — Resolve PR content from local sources

This step runs on both paths. No ADO connection required.

**Branch and work item:**
```bash
git symbolic-ref --short HEAD         # source branch
git ls-remote --heads origin dev      # verify target branch exists
```

If `dev` does not exist on the remote, ask: "What is the target branch for this PR?"

Extract the ADO work item ID from the branch name (pattern: `ADO-[0-9]+`).
If no ID found in the branch name, ask: "What is the ADO work item ID for this PR?"

**PR description:**
Check the current conversation for a PR description generated by `pr-describe`.
It will begin with a heading matching `## [ADO-ID] ...`.

- If found: extract the first heading line as the PR title, full markdown body as description.
- If not found: stop and output:
  > "No `pr-describe` output found in this conversation. Run `/pr-describe` first to
  > generate an ICEA-validated PR description, then run `/pr-create`."

**ICEA file:**
```bash
ls docs/icea/ADO-<ID>-*.md 2>/dev/null | head -1
```

Read it if found. Note "No ICEA found" if absent — the compliance check will handle this.

**Report scope:**
```
📋 PR scope
  Source  : <source-branch>
  Target  : dev  (or overridden value)
  ADO     : #<ID>
  ICEA    : docs/icea/ADO-<ID>-<name>.md  (or: not found)
  Title   : <derived from pr-describe output>
```

## Step 2 — ICEA compliance check

Always runs. No ADO connection required.

```bash
git diff dev..HEAD --stat
```

Invoke the `icea-review` skill:

```
Read .claude/plugin-path.txt to get PLUGIN_DIR (if absent, use §1a resolver), then
Read $PLUGIN_DIR/skills/icea-review/SKILL.md and run it against the current branch
diff and the ICEA document identified in Step 1. Produce a compact verdict only —
omit the full detail report at this stage.
```

**Based on the verdict:**

| Verdict | Action |
|---|---|
| ✅ READY FOR REVIEW | Proceed to Step 3 |
| ⚠️ NEEDS WORK BEFORE REVIEW | Carry warnings forward to Step 6 prompt — non-blocking |
| ❌ BLOCKED | Stop entirely. Do not proceed to findings gate or ADO prompt. |

**On BLOCKED:**

```
❌ PR CREATION BLOCKED — ICEA compliance check failed

  {list of Critical findings from icea-review}

Required action:
  Fix the above issues OR update the ICEA to cover them before creating the PR.

  To override (not recommended):
  Re-run with --skip-icea-check to bypass this gate.
  This will be noted in the PR description and the draft artifact.
```

**On `--skip-icea-check`:** proceed, carry a bypass note forward into both the
PR description and the draft artifact:
```
⚠️ ICEA compliance check bypassed by developer.
```

## Step 3 — Findings gate

Always runs. No ADO connection required.

Load `skills/shared/findings-gate.md` and execute the canonical bash functions
defined there verbatim. Do not re-implement the ledger-walking logic inline.

Collect:
- `cr_open`, `sec_open`, `ds_open` — open Critical/High FP IDs per ledger
- `total_open` — sum across all three ledgers
- `cr_dismissed`, `sec_dismissed`, `ds_dismissed` — accepted-risk dismissals

Carry all results forward to Step 6. Do not output anything here — the findings
summary is shown as part of the Step 6 ADO connection prompt.

**If `--skip-security-gate` is present:** mark the gate as bypassed, record the
developer's justification, carry forward to Step 6 and the draft artifact.

## Step 4 — Offline path (--offline flag only)

If `--offline` is set, skip Steps 5–7 entirely. Jump directly to Step 8.

`RELEASE_ID` and `SPRINT_ID` are already validated in the flag handling block above.

## Step 5 — Ask developer: connect to ADO or save draft?

Only reached if `--offline` is NOT set.

Display the compliance and findings summary first, then ask:

```
📋 Pre-submission summary
  ICEA compliance : ✅ READY FOR REVIEW  (or ⚠️ N warnings)
  Open findings   : N Critical/High across all ledgers  (or: ✅ None)

Connect to ADO and submit the PR, or save a PR draft for manual submission?

  1) Connect to ADO and submit
  2) Save PR draft only

Enter 1 or 2:
```

Wait for explicit input. Do not default to either option.

**If findings gate has `total_open > 0` and `--skip-security-gate` is NOT set:**
Display the findings gate warning block from `findings-gate.md` (`pr-create` variant)
above the prompt and require the developer to respond `yes / fix-first / no` before
the connect/draft question is shown.

- `fix-first` → stop, list FP IDs
- `no` → cancel entirely
- `yes` → carry acknowledgement forward, show connect/draft prompt

## Step 6 — Connect path: resolve PAT and ADO details

Only reached if developer chose option 1 in Step 5.

### PAT resolution

| Option | How | When to use |
|---|---|---|
| **A — Windows User Environment Variable** | `AZURE_DEVOPS_PAT` user env var | Default for all developers |
| **B — `.claude/settings.json`** | `{ "env": { "AZURE_DEVOPS_PAT": "..." } }` — must be gitignored | Acceptable for local dev only |
| **C — Paste at prompt** | Entered interactively | Emergency only |

```bash
echo $AZURE_DEVOPS_PAT
```

- If set and non-empty: use silently. Never echo or display the value.
- If not set: prompt the developer to paste it for this request only.

After Step 7 completes the variable is no longer referenced.
See `references/ado-api-guide.md` for PAT scope requirements.

### ADO connection details

```bash
git remote get-url origin
```

Parse `org`, `project`, `repo` from the remote URL:

| URL format | Example |
|---|---|
| `https://dev.azure.com/{org}/{project}/_git/{repo}` | Modern |
| `https://{org}.visualstudio.com/{project}/_git/{repo}` | Legacy |
| `https://{org}@dev.azure.com/{org}/{project}/_git/{repo}` | With username prefix |

If parsing fails: ask for org, project, and repo individually.

### Enrich PR title from ADO work item

```bash
curl -s --ssl-no-revoke \
  "https://dev.azure.com/{org}/{project}/_apis/wit/workitems/{ID}?api-version=7.1&\$select=System.Title" \
  -H "Authorization: Basic $(printf ':%s' "$AZURE_DEVOPS_PAT" | base64 -w 0)"
```

If the ADO title differs from the branch-derived title, use the ADO title.
If the call fails, keep the branch-derived title and note it in the output.

## Step 7 — Connect path: confirm and submit

Display confirmation and wait for explicit "yes":

```
About to create PR:

  From    : refs/heads/<source-branch>
  Into    : refs/heads/<target-branch>
  Org     : <org>
  Project : <project>
  Repo    : <repo>
  Title   : <title>
  WI      : ADO #<id>
  ICEA    : <verdict>
  Findings: <N open Critical/High — acknowledged>  (or: ✅ None)

Proceed? (yes / no)
```

If anything other than "yes": cancel.
> "PR creation cancelled. Your description is saved in this conversation — run
> `/pr-create` again when ready."

**Submit:**

```bash
curl -s --ssl-no-revoke -4 -w "\n%{http_code}" -X POST \
  "https://dev.azure.com/{org}/{project}/_apis/git/repositories/{repo}/pullrequests?api-version=7.1" \
  -H "Content-Type: application/json" \
  -H "Authorization: Basic $(printf ':%s' "$AZURE_DEVOPS_PAT" | base64 -w 0)" \
  -d '{
    "title": "{title}",
    "description": "{description}",
    "sourceRefName": "refs/heads/{source-branch}",
    "targetRefName": "refs/heads/{target-branch}",
    "workItemRefs": [{"id": "{work-item-id}"}]
  }'
```

Parse response:
- HTTP 201 → extract `.pullRequestId` — proceed to Step 9 (connect output)
- Any other status → extract `.message`, show error, suggest `references/ado-api-guide.md`

**On ADO unreachable (network error, non-HTTP failure):** do not hard fail.
Inform the developer and fall through to Step 8 (draft output) automatically:

```
⚠️ ADO submission failed — could not reach Azure DevOps.
   Saving PR draft artifact instead.
```

## Step 8 — Draft path: resolve output path and write artifact

Reached from: `--offline` flag (Step 4), developer chose option 2 (Step 5),
or ADO unreachable fallback (Step 7).

### Resolve Release ID and Sprint ID

**If `--offline`:** use `RELEASE_ID` and `SPRINT_ID` from arguments. Already validated.

**Otherwise:** prompt the developer. Both values are required — re-prompt if either is blank:

```
📋 PR draft output path
  Both values are required to write the artifact.

  Release ID : _
  Sprint ID  : _
```

On blank input for either:
```
⚠ Both Release ID and Sprint ID are required.
  Release ID : _
  Sprint ID  : _
```

Re-prompt until both are provided. Do not proceed with a partial path.

### Write artifact

Output path: `docs/Release<RELEASE_ID>/Sprint<SPRINT_ID>/pr-draft-ADO-<ID>.md`

```bash
mkdir -p "docs/Release<RELEASE_ID>/Sprint<SPRINT_ID>"
```

Artifact contents:

```markdown
# PR Draft — ADO #<ID>
_Generated: YYYY-MM-DD · Status: Pending submission_
_ICEA: <✅ READY FOR REVIEW | ⚠️ N warnings | ⚠️ Bypassed> · Findings gate: <✅ None | ⚠️ N acknowledged | ⚠️ Bypassed>_

---

## PR Details

| Field | Value |
|---|---|
| Title | [ADO-<ID>] <title> |
| Source branch | `<source-branch>` |
| Target branch | `dev` |
| Work item | ADO #<ID> |

> **Note:** Title derived from branch name — verify against ADO work item title before submitting.

---

## Submit URL

```
https://dev.azure.com/<org>/<project>/_git/<repo>/pullrequests/new
  ?sourceRef=<source-branch>
  &targetRef=dev
  &title=[ADO-<ID>]%20<url-encoded-title>
```

Open this URL in a browser — source branch, target branch, and title will be pre-filled.

---

## Description

<full PR description markdown from pr-describe output>

---

## Compliance

| Check | Result |
|---|---|
| ICEA compliance | <verdict> |
| Open Critical/High findings | <N across all ledgers, or: None> |

<If accepted-risk dismissals exist:>
### Accepted-risk dismissals (informational)
  [FP-xxxxxxxx] {VULN-TYPE} — {severity} — dismissed by {user} on {date}
  Reason: {justification}

<If any gate was bypassed:>
### Gate overrides
  ⚠️ <gate name> bypassed: <justification or "bypassed by developer">

---

## Manual submission steps

1. Open the Submit URL above
2. Paste the Description block into the ADO PR description field
3. Add reviewers
4. Link work item ADO #<ID> if not auto-linked
5. Submit
```

## Step 9 — Output result

**Connect path — success:**

```
✅ PR created successfully

   ID  : <pullRequestId>
   URL : https://dev.azure.com/{org}/{project}/_git/{repo}/pullrequest/<id>

Next: run /pr-spec-review spec=docs/icea/ADO-<ID>-<name>.md to validate the PR against its ICEA.
```

**Connect path — failure:** show HTTP status and `message` field from ADO response.
Do not retry. Suggest `references/ado-api-guide.md`.

**Draft path:**

```
📄 PR draft saved

   Path : docs/Release<RELEASE_ID>/Sprint<SPRINT_ID>/pr-draft-ADO-<ID>.md
   ICEA : <verdict>
   Findings: <summary>

   Submit URL pre-filled — open in browser to submit manually.
   When ready to submit via ADO: run /pr-create (PAT required).
```

## Hard rules

- NEVER store, echo, or log the PAT value at any point
- NEVER produce a PR artifact (submitted or draft) without running Steps 2 and 3 first
- NEVER skip the compliance check or findings gate — both run on both paths
- NEVER proceed past a BLOCKED icea-review verdict unless `--skip-icea-check` is present
- NEVER default the connect/draft question — wait for explicit developer input
- NEVER write the draft artifact with a partial path — both Release ID and Sprint ID required
- NEVER create the PR without explicit "yes" at Step 7 confirmation
- NEVER proceed if `pr-describe` has not been run — unvalidated descriptions must not be submitted
- If source branch equals target branch: abort — "Source and target branches are the same. Switch to your feature branch before creating a PR."
- If a PR already exists for this branch (HTTP 409): output the conflict message and stop — do not attempt to update the existing PR
- On ADO unreachable during submit: fall through to draft path automatically — do not hard fail
- ALWAYS note gate overrides (`--skip-icea-check`, `--skip-security-gate`) in both the PR description and the draft artifact
